SOLID is the most evaluated topic in design interviews, and the one where most answers are memorized acronyms with no example behind them.

The five letters are easy to recite. What separates a good answer is being able to show the **before and after** — the code that violates the principle, and the code that fixes it, and *why the fix is better*. That's what this post is.

**The basics** covers the five principles with an example each. **The advanced part** covers the complementary principles, the trade-offs, and where SOLID is genuinely wrong.

# Part 1 — The five principles

## S — Single Responsibility

> A class should have one, and only one, reason to change.

**Bad**: a `UserService` that validates input, saves to the database, sends a welcome email, and generates a PDF receipt. Four reasons to change: validation rules change, the schema changes, the email provider changes, the PDF layout changes. Four unrelated teams editing one file.

**Good**: separate the responsibilities.

```java
// before — four reasons to change
class UserService {
    void register(User user) {
        if (user.getEmail() == null) throw new ValidationException();
        db.save(user);
        smtp.send(user.getEmail(), "Welcome!");
        pdfGenerator.createReceipt(user);
    }
}

// after — each class has one job
class UserService {
    private final UserRepository repository;
    private final NotificationService notifications;

    void register(User user) {
        validator.validate(user);
        repository.save(user);
        notifications.sendWelcome(user);
    }
}
```

The subtlety worth mentioning in an interview: "one responsibility" doesn't mean "one method". The real formulation is **one reason to change** — or, more precisely, one *actor* who can request a change. Validation rules and PDF layouts come from different people, so they belong in different classes.

## O — Open/Closed

> Open for extension, closed for modification.

Adding a new case should not force you to edit existing code.

**Bad**: a `switch(type)` you have to open every time a new type appears.

```java
class FeeCalculator {
    double calculate(Order order) {
        switch (order.getType()) {
            case STANDARD: return order.getTotal() * 0.02;
            case PREMIUM:  return order.getTotal() * 0.01;
            case ENTERPRISE: return 0;
            // every new customer type = editing this method again
        }
    }
}
```

**Good**: Strategy and polymorphism.

```java
interface FeePolicy {
    double calculate(Order order);
}

class StandardFee implements FeePolicy {
    public double calculate(Order o) { return o.getTotal() * 0.02; }
}

class PremiumFee implements FeePolicy {
    public double calculate(Order o) { return o.getTotal() * 0.01; }
}

class FeeCalculator {
    private final FeePolicy policy;

    FeeCalculator(FeePolicy policy) { this.policy = policy; }

    double calculate(Order order) { return policy.calculate(order); }
}
```

A new customer type is now a **new file**, not an edit to a tested one. The existing classes don't get recompiled, retested, or re-reviewed.

The mechanism underneath is four ideas working together:

- **Composition** — a class holds another class and delegates a task to it.
- **Interfaces** — I don't care about the concrete class, only the contract.
- **Polymorphism** — that object can be anything implementing the interface.
- **Strategy** — I use that to swap the algorithm without touching the rest of the code.

## L — Liskov Substitution

> If a class inherits from another, it must be usable anywhere the parent is expected, without breaking the program's expected behavior.

This is the one people struggle to explain, and it has the clearest example of the five.

**Correct:**

```java
class Animal {
    void makeSound() {}
}

class Dog extends Animal {
    @Override
    void makeSound() {
        System.out.println("Woof");
    }
}

void listen(Animal animal) {
    animal.makeSound();
}

listen(new Dog());   // ✅ works correctly
```

`Dog` can **replace** `Animal` because it behaves the way the calling code expects.

**Incorrect:**

```java
class Bird {
    void fly() {}
}

class Penguin extends Bird {
    @Override
    void fly() {
        throw new UnsupportedOperationException();
    }
}

void makeFly(Bird bird) {
    bird.fly();
}

makeFly(new Penguin());   // ❌ breaks the program
```

`Penguin` **cannot replace** `Bird`, because the code expects that any `Bird` can fly. The class hierarchy says a penguin is a bird — which is true biologically and false for this program, because the hierarchy was built around flying.

> A subclass must never break the expected behavior of the parent class. If code works with the parent, it must also work with any subclass.

The fix is not a clever cast. It's admitting the hierarchy was wrong: split `Bird` into `Bird` and `FlyingBird`, or drop inheritance and give flying its own interface.

**How LSP gets violated in practice**, beyond throwing exceptions:

- The subclass **strengthens a precondition** — the parent accepts any integer, the subclass demands a positive one.
- The subclass **weakens a postcondition** — the parent guarantees a sorted result, the subclass doesn't.
- The subclass **changes the semantics** — `setWidth` on a `Rectangle` sets one dimension; on a `Square` subclass it has to set both, and any code that sets width then reads height gets a surprise. This is the textbook example, and it's a real one.

The signal to watch for: a subclass that overrides a method to do nothing, or to throw. That's LSP telling you the inheritance is fake.

## I — Interface Segregation

> Many small, specific interfaces beat one giant one.

Don't force a class to implement methods it doesn't use.

```java
// bad — every implementer must handle all of it
interface Worker {
    void work();
    void eat();
    void attendMeeting();
}

class Robot implements Worker {
    public void work() { /* ... */ }
    public void eat() { throw new UnsupportedOperationException(); }   // ← LSP violation too
    public void attendMeeting() { throw new UnsupportedOperationException(); }
}

// good — separate capabilities
interface Workable { void work(); }
interface Feedable { void eat(); }

class Robot implements Workable { public void work() { /* ... */ } }
class Human implements Workable, Feedable { /* ... */ }
```

Notice that the fat interface **forced** an LSP violation. These principles are not independent — violating one usually drags another with it.

Go builds this into the culture: `io.Reader` has exactly one method, and half the standard library speaks it. Small interfaces compose; large ones only get implemented once.

## D — Dependency Inversion

> Depend on abstractions, not on concrete implementations.

More precisely: **high-level modules should not depend on low-level modules; both should depend on abstractions.**

```java
// bad — the service creates and depends on a concrete class
class UserService {
    private final PostgresUserRepository repo = new PostgresUserRepository();
}

// good — depends on a contract, receives it from outside
class UserService {
    private final UserRepository repo;

    UserService(UserRepository repo) {   // constructor injection
        this.repo = repo;
    }
}
```

This is the foundation of **dependency injection** and of testing with mocks. In the second version you can pass an in-memory fake repository and unit-test the business logic with no database, no Docker, and no test that takes four seconds.

The "inversion" in the name is about the direction of the dependency arrow. Normally the business logic would point at the database. With DIP, the business logic defines the interface it needs, and the database implementation points **at that interface**. The domain stops depending on infrastructure; infrastructure starts depending on the domain.

This is the same idea that Hexagonal Architecture and Clean Architecture are built on, scaled up to a whole application.

# Part 2 — Advanced

## The sentences worth having ready

These are the phrases that carry the most weight in an interview, because each one compresses a lot of judgment:

- **"Prefer composition over inheritance."** It's the principle behind half of the design patterns.
- **"Program against interfaces, not implementations."** The basis of dependency injection and testing.
- **"A pattern is an answer to a problem; without the problem, it's over-engineering."** Show judgment, not a catalog.
- **"Make code easy to delete and to change, not just to reuse."** Low coupling beats premature reuse.
- **"Code is read many more times than it's written."** Clarity over cleverness.

That third one matters more than the others. Interviews reward people who know when *not* to apply a principle.

## The complementary principles

SOLID is not the whole list. These come up just as often:

**DRY — Don't Repeat Yourself.** Don't duplicate logic. The trap: don't over-abstract. Two lines that look similar are not always real duplication — if they change for different reasons, merging them creates a coupling you'll regret. Duplication is cheaper than the wrong abstraction.

**KISS — Keep It Simple.** The simplest solution that works, not the cleverest.

**YAGNI — You Aren't Gonna Need It.** Don't build for imaginary future requirements. The plugin architecture for the second payment provider that never arrives is pure cost.

**Composition over inheritance.** Inheritance couples tightly and is rigid; composition (has-a) is flexible and testable.

**Law of Demeter.** Talk only to your neighbors. Avoid `a.getB().getC().doThing()` — a class should communicate with its direct dependencies without knowing the internal structure of other objects, to reduce coupling and make future changes cheaper.

**High cohesion, low coupling.** Each module does one thing well and depends little on the others. This is what all of the above is actually *for*.

## Where the principles conflict

Nobody tells you this part, and it's the most useful thing to understand.

**DRY vs Single Responsibility.** Deduplicating code that two classes share can force both to depend on a shared module, coupling things that should have stayed separate. Sometimes the right answer is to leave the duplication.

**Open/Closed vs YAGNI.** Making everything extensible means building abstraction points for extensions that may never exist. Every `interface` with one implementation is a small tax. The honest rule: **write it concretely the first time, abstract on the second or third**, when you can see the actual axis of variation.

**Interface Segregation vs simplicity.** Splitting one interface into six makes each one clean and the overall system harder to navigate.

The resolution is always the same: these are heuristics for reducing coupling and increasing cohesion. When two of them disagree, go back to the actual goal and ask which choice makes the code easier to change. Not which principle has a nicer name.

## How to apply this without over-engineering

A practical order of operations:

1. **Write it directly.** No interfaces, no strategies. Get it working and understood.
2. **When the second case arrives**, look at what varies. That's your extension point — and now you know its real shape instead of guessing.
3. **Extract the abstraction** at that seam.
4. **Stop.** Don't extract the third layer for the case that hasn't happened.

Most over-engineered code comes from applying step 3 before step 2 exists.

## The strongest interview answer

If someone asks "do you follow SOLID?", the memorized answer is to list the five letters. The better answer is roughly:

> "I use them as diagnostics rather than rules. If a class is hard to test, it's usually a Dependency Inversion problem. If a small change forces edits in five files, it's Single Responsibility. If I keep editing a `switch`, it's Open/Closed. They're most useful for naming a smell I already feel, not for designing upfront."

That says you've applied them to real code, which is the thing being tested.

## Where SOLID is criticized

Worth knowing, because a senior interviewer might raise it:

- **It's from 2000, and object-oriented.** In a codebase built from pure functions and immutable data, several principles are either automatic or don't apply.
- **Open/Closed can be an argument for speculative generality.** Taken literally, it says never modify existing code — which produces layers of abstraction defending against changes nobody requested. Modern practice with good tests and version control is comfortable modifying code.
- **Single Responsibility is vague.** "One reason to change" depends entirely on how you frame a reason. It's a good instinct and a poor definition.
- **They can conflict**, as covered above, and the acronym gives you no way to break the tie.

None of that makes them worthless. It makes them heuristics rather than laws, which is how experienced people use them anyway.

---

## The short version

1. **S** — one reason to change, measured by *who* asks for the change.
2. **O** — new behavior should be a new file, not an edit to a tested one.
3. **L** — if a subclass throws or no-ops on an inherited method, the hierarchy is wrong.
4. **I** — a fat interface forces LSP violations downstream.
5. **D** — depend on the contract; it's what makes the code testable.

And the one that matters most: **a principle without a problem is over-engineering.**
