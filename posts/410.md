# Part 1 — The basics

## The vocabulary

- **Class** — a structure that defines objects and their behavior. The blueprint.
- **Object** — an instance created from a class. The thing itself.
- **Constructor** — a method used to initialize an object.
- **Method** — a function that belongs to a class.
- **Attribute / field** — a variable that belongs to a class.
- **Interface** — a contract defining which methods a class must implement.
- **Instance** — one concrete object built from a class.

The class is the recipe; the object is the cake. There is one recipe and there can be a thousand cakes, each with its own state.

## What OOP is actually for

The honest answer, and a good one to give in an interview: **OOP is a way to manage complexity by bundling state with the operations allowed on that state.**

Before OOP, you had data structures and a pile of functions that could do anything to them. The value of an object is that it owns its own data and controls how that data changes. Everything else — inheritance, polymorphism — is machinery built on top of that idea.

## Pillar 1 — Encapsulation

Hide the internal state and expose a controlled API.

In practice: `private` fields plus methods that control access. The point isn't the getters and setters — it's that **invalid states become unreachable**.

```java
public class Account {
    private BigDecimal balance;

    public void withdraw(BigDecimal amount) {
        if (amount.compareTo(balance) > 0) {
            throw new InsufficientFundsException();
        }
        balance = balance.subtract(amount);
    }
}
```

The balance cannot go negative, because there is no way to reach it except through a method that refuses. If `balance` were public, that guarantee would live in every caller instead — and one of them will get it wrong.

Access modifiers, using Java as the reference:

| Modifier | Visible from |
| --- | --- |
| `public` | Everywhere |
| `protected` | Class + subclasses + same package |
| *(none)* | Same package |
| `private` | The class only |

Go does the same thing with a capitalization rule; Python does it with a naming convention and no enforcement at all. Different mechanisms, same idea.

**The rule**: expose the minimum necessary. And a getter/setter pair for every private field is not encapsulation — it's a public field with more typing.

## Pillar 2 — Inheritance

A class `extends` another and inherits its fields and methods. **A Car IS A Vehicle.**

```java
class Animal {
    void makeSound() {}
}

class Dog extends Animal {
    @Override
    void makeSound() {
        System.out.println("Woof");
    }
}
```

Java has **single inheritance** — one parent class only. `super` reaches the parent's implementation.

Inheritance is the pillar that gets overused. It's genuinely useful when there's a real "is-a" relationship *and* the subclass can stand in for the parent everywhere. When either of those isn't true, you want composition instead — which is the whole second half of this post.

## Pillar 3 — Polymorphism

The same method behaves differently depending on the object's real runtime type.

```java
Vehicle v = new Car();
v.accelerate();      // runs Car's version, not Vehicle's
```

The variable is declared as `Vehicle`, but the object is a `Car`, and the `Car` implementation runs. This is **dynamic dispatch**, and it's what makes "program against interfaces" possible at all.

Two kinds, and interviewers do ask for the distinction:

- **Runtime polymorphism** — overriding plus dynamic dispatch. Which method runs is decided while the program is running.
- **Compile-time polymorphism** — overloading and generics. Which method runs is decided by the compiler.

And the related pair:

- **Overload** — same method name, different parameters. Resolved at compile time.
- **Override** — a subclass redefines a parent method. Resolved at runtime.

## Pillar 4 — Abstraction

Expose the **what**, hide the **how**.

A `PaymentProcessor` interface says a payment can be processed. It says nothing about Stripe, retries, or webhook signatures. The caller works with the idea; one implementation deals with the reality.

Abstraction is what lets you change the "how" without touching any caller. It's also the pillar most easily overdone — an abstraction with exactly one implementation and no prospect of a second is usually just an extra file.

## Interface vs abstract class

The classic question. The short answer:

| | Interface | Abstract class |
| --- | --- | --- |
| State (fields) | No (constants only) | Yes |
| Constructors | No | Yes |
| Implemented methods | Default methods only | Yes, freely |
| How many | A class implements many | A class extends one |
| Answers | "what can this do?" | "what do these share?" |

**Use an interface** for a contract or capability that many unrelated classes can fulfill. `Comparable`, `Serializable`, `PaymentProcessor`.

**Use an abstract class** to share base code between closely related classes that genuinely form a family.

When in doubt, interface. It's the looser coupling, and you can always add an abstract base class implementing it later. The reverse is much harder.

## Composition

Instead of inheriting behavior, a class **holds** another object and delegates to it. **A Car HAS AN Engine.**

```java
class Engine {
    void start() {}
}

class Car {
    private final Engine engine;

    Car(Engine engine) {
        this.engine = engine;
    }

    void turnOn() {
        engine.start();     // delegates the work to the engine
    }
}
```

Composition is when a class uses another class to perform a task, instead of inheriting from it. That makes the code more flexible, more reusable, and easier to maintain.

# Part 2 — Advanced

## Composition over inheritance

This is the single most valuable sentence you can say in an OOP interview, and the principle behind half of the design patterns.

- **Inheritance**: a class **is** another class and inherits its characteristics and behaviors.
- **Composition**: a class **has** another class inside it and uses its functionality rather than inheriting it.

Here's the concrete case that makes it click.

### Using inheritance — not a good idea

```java
class MySQLDatabase {
    void saveUser(User user) {
        System.out.println("Saving to MySQL");
    }
}

class UserService extends MySQLDatabase {
    void register(User user) {
        saveUser(user);
    }
}
```

It works. Then the company moves to PostgreSQL, and now you have to change the inheritance itself:

```java
class UserService extends PostgreSQLDatabase { ... }
```

`UserService` is welded to a specific database. Worse, it has silently inherited every public method of `MySQLDatabase` into its own API — callers can now reach past your service straight into persistence.

### Using composition — the recommended version

```java
interface Database {
    void saveUser(User user);
}

class MySQLDatabase implements Database {
    public void saveUser(User user) {
        System.out.println("Saving to MySQL");
    }
}

class PostgreSQLDatabase implements Database {
    public void saveUser(User user) {
        System.out.println("Saving to PostgreSQL");
    }
}

class UserService {
    private final Database database;

    UserService(Database database) {
        this.database = database;
    }

    void register(User user) {
        database.saveUser(user);
    }
}
```

Usage:

```java
Database db = new MySQLDatabase();
UserService service = new UserService(db);
service.register(new User());
```

And if you switch databases tomorrow:

```java
Database db = new PostgreSQLDatabase();
UserService service = new UserService(db);
```

**Not one line of `UserService` changed.**

### Why this version is better

`UserService` only handles business logic — registering users. It does not know **how** the data is stored. That responsibility belongs to the database implementation.

In other words:

- `UserService` **has a** `Database` → composition.
- `MySQLDatabase` and `PostgreSQLDatabase` **implement** the `Database` interface.
- If you swap the implementation, `UserService` keeps working identically.

That's exactly the idea behind "prefer composition over inheritance". In real Spring Boot or Go projects, this shape appears constantly through **dependency injection**. It also makes the service trivially testable: pass in an in-memory fake `Database` and you have a unit test with no database at all.

## The three ideas that work together

The composition example above is really four principles stacked:

- **Composition** — I have an object and delegate to it.
- **Interfaces** — I don't care about the concrete class, only the contract.
- **Polymorphism** — that object can be anything implementing the interface.
- **Strategy** — I exploit that to change behavior without touching the rest of the code.

If you can explain how those four connect, you've answered most OOP interview questions in advance.

## Program against interfaces, not implementations

> Don't depend on a specific class; depend on a contract.

This is the foundation of dependency injection and of testing with mocks. The dependency you declare should almost always be an interface, not a concrete class.

The practical test: can you replace this dependency with a fake in a test without changing the class under test? If not, you're coupled to an implementation.

## The Law of Demeter

Talk only to your immediate neighbors. Avoid `a.getB().getC().doThing()` — the "train wreck".

A class should communicate only with its direct dependencies, without knowing the internal structure of other objects. Every `.` in that chain is a fact about someone else's internals that you've now hardcoded — and any of them changing breaks you.

```java
// coupled to three levels of structure
order.getCustomer().getAddress().getCity();

// ask, don't traverse
order.getShippingCity();
```

## High cohesion, low coupling

The two-word summary of good design:

- **Cohesion** — how focused a module is. High cohesion means everything in the class belongs to one job.
- **Coupling** — how much modules depend on each other. Low coupling means changing one doesn't cascade.

Every principle in this post and the SOLID one is, ultimately, a specific technique for raising cohesion or lowering coupling.

## Object equality and identity

The distinction that causes real bugs:

- **Identity** — are these the same object in memory? (`==` in Java)
- **Equality** — do these represent the same thing? (`.equals()`)

Two `User` objects loaded separately from the database are different objects with equal content. If you compare them with `==` you'll get `false` and a confusing afternoon.

In Java, **`equals` and `hashCode` must be implemented together and stay consistent**: equal objects must return equal hash codes. Break that and your `HashSet` will hold duplicates, silently.

## Immutability

An immutable object cannot change after construction. It's underrated and it solves several problems at once:

- It's automatically thread-safe — no shared mutable state, no race conditions.
- It's safe to use as a map key or in a set, because its hash can't change under you.
- It can't be modified by a method you passed it to.

```java
public record Money(BigDecimal amount, Currency currency) {}
```

Java records, Kotlin data classes, Go structs passed by value — all leaning the same direction. The rule of thumb: **make things immutable unless you have a reason not to.**

## The SOLID connection

The four pillars tell you what OOP *is*. SOLID tells you how to use it without making a mess. The five principles map directly onto everything above:

- **S** — Single Responsibility: cohesion.
- **O** — Open/Closed: polymorphism instead of `switch`.
- **L** — Liskov Substitution: inheritance done honestly.
- **I** — Interface Segregation: small contracts.
- **D** — Dependency Inversion: program against interfaces.

Each one is a specific technique for keeping the four pillars from turning into a mess.

## Common criticisms worth knowing

Being able to argue against OOP is a stronger signal than defending it uncritically.

- **Deep hierarchies are brittle.** A change at the root ripples through every descendant, and behavior gets scattered across five files you have to read simultaneously.
- **Not everything is an object.** A pure calculation is a function. Wrapping it in a `CalculatorManagerFactory` adds ceremony and removes clarity.
- **Mutable shared state is the hard part of concurrency**, and classic OOP encourages it. This is a real reason functional ideas — immutability, pure functions — have been absorbed into every modern OO language.
- **Go left inheritance out entirely** and is doing fine. That's evidence the pillar was optional.

Use objects where state and behavior genuinely belong together. Use functions where they don't.

---

## The short version

1. **Encapsulation is about making invalid states unreachable**, not about generating getters.
2. **Composition over inheritance.** The database example is the one to have ready.
3. **Program against interfaces** — it's what makes code testable.
4. **Interface for a capability, abstract class for a family.** When in doubt, interface.
5. **Immutable by default.** It removes an entire category of bugs.
