## Explaining a Bug — Root Cause, Quick Fix vs. Real Fix

## Introduction

Step 11 of the plan from *"How I'm training myself to stop sounding basic in english meetings"*.

This one happens under pressure. Something is broken, people are waiting, and I have to explain in English what went wrong — usually to someone who won't understand the technical part and only wants to know when it's fixed.

---

## Why This Matters

When I explain a bug badly in English, I do it in the wrong order.

I start where I started: the symptom, then the investigation, then the dead ends, then what I found. Chronological. Which is how I *experienced* it, and the worst possible way to *tell* it — because the person listening is thinking "is it fixed?" through the entire story.

There's also a second problem, and it's not English at all. Under stress I explain defensively — proving it wasn't my fault before saying what actually happened. That reads badly in every language.

---

## The Core Structure

```
[Impact] + [Cause] + [Status] + [Next]

Checkout has been failing for about 10% of users since the
Tuesday deploy. The cause is a null country code for accounts
created before 2023. I've pushed a fix to staging; it goes to
production in an hour. Then I'll add validation so it can't
happen again.
```

- **Impact** — who is affected and how badly. Not the code, the *people*.
- **Cause** — one sentence. The technical detail can come later if they ask.
- **Status** — fixed, being fixed, or still investigating.
- **Next** — the real fix, or what you need.

Answer "how bad is it and is it fixed" in the first two sentences. Everything after that is optional detail.

## Situations & Examples

### Situation 1: Reporting a Bug You Just Found

**Context:** Production is broken and you're the one telling people.

**Bad way:**
```
So I was looking at the logs and I noticed something weird
with the payment service, and I checked the database and then
I found that maybe the migration...
```
❌ Three sentences in and nobody knows if customers are affected.

**Good way:**
```
Heads up: payments are failing for new signups since about
2pm. Roughly 40 users affected so far. I'm on it — I think
it's the migration from this morning, still confirming.
```
✅ Impact first, scale, status. Details on request.

---

### Situation 2: Explaining the Root Cause

**Context:** It's fixed. Now they want to know what happened.

**Bad way:**
```
There was a problem with the database.
```
❌ Too vague to learn anything from.

**Good way:**
```
Root cause: the migration added a NOT NULL column without a
default. Existing rows got an empty string, and the payment
validator rejects empty strings. So it only affected accounts
that existed before the migration ran.
```
✅ The chain, in order, in four sentences. Anyone can follow it.

**Explaining it as a chain** — "X caused Y, which caused Z" — is what makes a root cause understandable to someone who wasn't debugging it.

---

### Situation 3: Quick Fix vs. Real Fix

**Context:** You can stop the bleeding in ten minutes, or fix it properly in two days. This decision needs to be explicit and it isn't yours alone.

**Bad way:**
```
I fixed it.
```
❌ Which one did you do? Nobody knows there's still a problem.

**Good way:**
```
Two options. Quick fix: I disable that validation rule, ten
minutes, payments work again tonight — but bad data keeps
getting in. Real fix: backfill the column and add a
constraint, about a day, and I'd need staging access.

My suggestion is do the quick fix now and the real one this
week. Sound okay?
```
✅ Both options, both costs, a recommendation. They decide, informed.

**Never let a quick fix disappear.** Say out loud that it's temporary and put the real one somewhere visible.

---

### Situation 4: Explaining to a Non-Technical Person

**Context:** Your manager, a PM, a client. They don't want the stack trace.

**Bad way:**
```
The webhook handler wasn't idempotent so retries were
double-inserting rows.
```
❌ Accurate and completely useless to them.

**Good way:**
```
Short version: some customers were charged twice. It happened
when our payment provider retried a request and we counted it
as a new one. It's fixed, and I have a list of the 12 affected
customers for refunds.
```
✅ In their terms, with the action they actually care about.

---

### Situation 5: You Still Don't Know What's Wrong

**Context:** An hour in, no cause found, and someone is asking for an update.

**Bad way:**
```
I'm still looking into it.
```
❌ Says nothing. They'll ask again in fifteen minutes.

**Good way:**
```
Still investigating. What I know: it started after the 2pm
deploy and only affects EU users. I've ruled out the database
and the cache. Next I'm checking the CDN config. I'll update
in 30 minutes either way.
```
✅ What you know, what you've eliminated, what's next, when you'll speak again.

**Ruling things out is progress.** Say it that way.

---

### Situation 6: You Caused the Bug

**Context:** It was your PR. This is the one where my English gets worst, because I'm defending before explaining.

**Bad way:**
```
Well, the ticket didn't say anything about that case, and
nobody mentioned it in review, so...
```
❌ Everyone can hear the defence and now nobody trusts the explanation.

**Good way:**
```
This one's mine — my change last week didn't handle the empty
list. Fix is up, it's a two-line guard. I'm also adding a test
for that case, because review wouldn't have caught it either.
```
✅ Own it in one sentence, then move to the fix. Nobody cares beyond that.

## Phrases

### Reporting impact first

```
Heads up: X is broken for Y users.
We have an issue in production — checkout is failing.
This is affecting about 10% of requests.
Not urgent, but worth flagging: ...
Customer-facing, so we should move on it.
```

### Describing the cause

```
Root cause is...
It turned out to be...
The problem is that X doesn't handle Y.
It only happens when...
The chain is: X causes Y, which breaks Z.
It's a race condition between A and B.
```

### Status updates

```
Still investigating. Here's what I know so far.
I've ruled out the database and the cache.
Fix is in review.
Deployed to staging, going to production in an hour.
It's fixed. Monitoring for the next hour.
```

### Quick fix vs. real fix

```
There's a quick fix and a proper fix.
Short term I can do X, which takes ten minutes.
The real fix is bigger — about a day.
I'd suggest the quick one now and the proper one this week.
This is a workaround, not a solution. Let's track it.
```

### For non-technical people

```
Short version: ...
In practical terms, that means...
For the user, what happened was...
The impact is X customers, and here's what we do about it.
Nothing else is affected.
```

### Owning it

```
This one's on me.
My change caused this.
I missed that case.
That was my mistake — fix is up.
```

### Preventing it next time

```
I'm adding a test for that case.
We should add an alert for this.
This would have been caught by X, so I'm adding it.
Worth a post-mortem, or is that overkill?
```

---

## Key Rules

1. **Impact first, always**
   - Who's affected and how badly, before any technical detail.
2. **Cause in one sentence**
   - Expand only if they ask.
3. **Explain it as a chain**
   - "X caused Y, which caused Z." Not a list of clues.
4. **Say explicitly which fix you did**
   - Quick fix and real fix are different things. Name the one you shipped.
5. **Never let a workaround go untracked**
   - If you don't say it's temporary, it becomes permanent.
6. **Ruling things out is a status update**
   - "I've eliminated the DB and cache" is real progress. Report it.
7. **Give a time for the next update**
   - "I'll update in 30 minutes" stops five people from asking.
8. **Own it in one sentence, then move on**
   - Defending yourself takes longer and works worse.

---

## What NOT to Do

| ❌ Don't say | ✅ Say instead |
|---|---|
| *Chronological story of your debugging* | "Impact first, then cause." |
| "There's a problem with the database." | "Root cause: the migration added a NOT NULL column." |
| "I fixed it." | "Quick fix is live. Real fix needs a day — I'd do it this week." |
| "I'm still looking into it." | "Ruled out the DB and cache. Checking the CDN. Update in 30 min." |
| "The ticket didn't say..." | "This one's mine. Fix is up." |
| *Stack trace to your manager* | "Short version: 12 customers were charged twice." |
| *Silent workaround* | "This is a workaround — let's track the real fix." |

---

## The Bigger Picture

Explaining a bug well is mostly about **order**, not vocabulary. Impact, cause, status, next. Four things, always in that order, and half the English difficulty disappears because I already know what sentence comes next.

The other half is resisting the urge to explain how hard the debugging was. Nobody is grading the investigation. They want to know if it's fixed.
