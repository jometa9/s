## Understanding the Daily Standup & Sprint
 
### Context: What Is a Daily Standup?
 
Most developers who work remotely have a meeting, commonly called "the daily." In this daily meeting, the developer team, tech lead, manager, and sometimes other roles (like the product owner) gather together.
 
The meeting starts with a greeting from the team. Most of the time, cameras are turned off to avoid internet lag—although honestly, in most cases, the internet instability is just an excuse. Maybe you're in your pajamas and don't want your team to see you in that cozy hoodie with minions on it. 
 
### Sprint & Role of the Moderator
 
Depending on the sprint (a work methodology that divides many tasks with objectives to complete in two weeks, generally), the sprint has a moderator. This moderator is the owner of the daily meeting and documentation about the sprint. Their responsibilities include:
 
* **Share the board every day** in the meeting showing the Jira or Kanban board with all team tasks.
* **Moderate the time** when each team member speaks to explain their task (if they have blockers, need support, or if everything is going well).
* **Keep documentation** about the epic, history, tasks, and subtasks to complete in this sprint—for example:
  * How many hours each developer logged for each task
  * A comment on the Jira task with pull requests and evidence of task completion
**Quick note:** The sprint workflow can change depending on the team and organization.

## Good vs. Poor Standups — Real Examples
 
### What NOT to Do — Examples of Poor Standups
 
Here are six standups with problems. Notice what makes them unclear:
 
**Example 1 — Buried information, needs alignment:**
```
Morning, I worked to complete task 437. All the code is finished, 
but today I'm adding unit tests. I maybe need help from Martin 
to coordinate how we write the tests so we're aligned with the epic.
```
**Problem:** "All code is finished" vs. "today I'm adding tests" — which is it? 
**Better structure:** Yesterday (done) → Today (doing) → Blocker (what you need help with).
 
---
 
**Example 2 — Vague, no timeline:**
```
Hi, I'm currently working on the API for external client integration. 
The basic structure and logic are done. I'm having trouble understanding 
the flow of the API I'm using. For now, I'll keep working on it and 
let you know if I have any questions.
```
**Problem:** "Let you know if I have questions" is passive. What's your blocker *right now*?
**Better:** Flag the blocker immediately. "I need 30 mins to review the docs with someone."
 
---
 
**Example 3 — Too many side tasks:**
```
How are you? I'm continuing without VPS credentials to connect to 
the repository to download the new code, so I'm waiting for that. 
But I'm working helping Diego with the onboarding and documenting 
the frontend architecture on Confluence.
```
**Problem:** It's unclear what your main task is. Too many things happening.
**Better:** List priorities. "Blocker: VPS credentials. In the meantime, I'm helping Diego."
 
---
 
**Example 4 — Admin talk mixed with status:**
```
Hi, I forgot yesterday to add my hours to the task, but all of this 
is done. Today I'll make the documentation and the pull request to 
move to testing and continue with the next task.
```
**Problem:** Don't mention logging hours in a standup (that's admin, not technical status).
**Better:** Focus only on what's done, doing, and blocked.
 
---
 
**Example 5 — Question buried mid-update:**
```
Morning, yesterday at the end of the day I discovered a corner case 
without coverage that needs attention. What do I do with this? Create 
a bug ticket or something like that? It's about error handling on the 
frontend—missing many cases. As for the rest, I'll continue with my 
task. I estimate that tomorrow I'll have everything in order to make 
the PR and close it.
```
**Problem:** You're asking for decision-making in standup instead of deciding yourself.
**Better:** "Found corner case. Logging it as a bug ticket. Continuing task. PR ready tomorrow."
 
---
 
**Example 6 — External blocker, personal info:**
```
My internet is cutting out, sorry about this. After the meeting, 
I'm going to the coworking space with stable internet. As for the 
status of my tasks, I closed the pull request and documentation. 
Today I'm starting the new requirement. I'm asking for help if 
I don't understand something about the task, but for now, all is okay.
```
**Problem:** Personal updates ("my internet") + asking vague help + "all is okay" (too casual).
**Better:** Keep personal brief, flag specific blockers.

### What TO Do — Real-World Good Standups
 
**Same scenarios, fixed:**
 
**1. Task complete, needs alignment:**
```
Yesterday: Finished task 437—all code is done.
Today: Adding unit tests.
Blocker: Need Martin's help to align our test strategy with the epic.
Can we sync 30 mins?
```
 
**2. Stuck on external API:**
```
Yesterday: Built basic structure for external client API.
Today: Debugging the API flow.
Blocker: Need clarification on the authentication flow. 
I'll reach out if I get stuck.
```
 
**3. Waiting on credentials:**
```
Yesterday: Blocked on VPS credentials.
Today: Helping Diego with onboarding. Documenting frontend 
architecture on Confluence.
Blocker: Still waiting on VPS access. Can't update code yet.
```
 
**4. Done, moving forward:**
```
Yesterday: Completed task—PR and documentation done.
Today: Starting new requirement.
No blockers. Ready to go.
```
 
**5. Found a bug:**
```
Yesterday: Found uncovered edge case in error handling (frontend).
Today: Logging it as a bug. Continuing my task.
No blockers. PR ready by tomorrow.
```
 
**6. Moving locations:**
```
Yesterday: Closed PR and docs.
Today: Starting new requirement. Moving to coworking space 
for stable internet.
Blocker: Resolved (by changing location).
```

### Key Rules for Standups
 
1. **Use past → present → blocker structure** (Yesterday → Today → What's in your way?)
2. **Be specific:** "working on stuff" |  "finished the payment API endpoint"
3. **Keep it 30–60 seconds** — if you need 2 minutes, save it for a 1:1 with your manager
4. **Flag blockers early** — don't wait until end of day
5. **No small talk** — save "how are you?" for after the meeting
6. **Estimate if unclear** — "Should have this done by Thursday" or "Need 2 days"
7. **Ask for specific help** — not "let me know" but "can I grab 30 mins with Sarah?"

### Real-World Standup Examples (Messy But Clear)
 
**Example A — Honest & conversational:**
```
Okay, so yesterday I finished the user auth refactor. Got the 
tests passing. Today I'm starting the database migration. I'm 
a little worried about downtime, but I think we're good. No 
blockers right now.
```
 
**Example B — Rambling but recovers:**
```
Um, so I was working on the mobile payment thing, and there's 
like... it's integrating with Stripe, and the webhook wasn't 
firing right. But I figured it out last night. Should have it 
working today. Might need QA to test before we merge though.
```
 
**Example C — Real blocker:**
```
Finished the caching layer yesterday. Today's the API endpoint. 
But I'm still waiting on the database schema from the other team—
they said Monday. Can't really move forward without that.
```
 
**Example D — Asking for help (direct):**
```
So the frontend tests are failing in a weird way. I've been 
looking at it for like an hour. Sarah, you worked with Cypress 
last sprint, right? Can I grab 15 mins later?
```
 
---
 
## Key Takeaway
 
A good standup is:
- **Clear** (Yesterday → Today → Blocker)
- **Specific** (names, ticket numbers, what you're stuck on)
- **Short** (under 60 seconds)
- **Honest** (don't hide blockers)
The goal is NOT to impress anyone. It's to **keep the team unblocked and aligned**.
