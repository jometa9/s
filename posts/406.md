# Part 1 — The basics

## Why Go

- **Cheap concurrency.** Goroutines are managed by the Go runtime, not by the operating system. You can run thousands of them; you cannot run thousands of OS threads.
- **Trivial deploys.** The compiler produces a single static binary. No runtime to install on the server.
- **Errors at compile time.** Static typing catches most of the mistakes before the program runs.
- **Errors as values.** There is no `try/catch`. A function that can fail returns an `error` as its last return value, and you handle it right there.

That last point is the one that surprises people coming from Java or JavaScript. In Go, error handling is not a separate control flow — it's just code.

```go
user, err := repo.FindByID(id)
if err != nil {
    return nil, err
}
```

You will write that block hundreds of times. That's the deal.

## The standard library you'll actually use

| Package | What it's for |
| --- | --- |
| `net/http` | HTTP servers and clients |
| `encoding/json` | Structs ↔ JSON (marshal / unmarshal) |
| `context` | Timeouts, cancellation, request-scoped values |
| `sync` | Synchronizing goroutines (Mutex, RWMutex, WaitGroup, Once) |
| `database/sql` | Standard API for SQL databases — needs a driver |
| `errors` | Creating and wrapping errors (`New`, `Is`, `As`) |
| `fmt` | Printing and formatting strings |
| `time` | Dates, durations, timers |
| `os` | Files, environment variables, processes |
| `io` | Reading and writing data (`io/ioutil` is deprecated) |

One note on `database/sql`: it does not talk to the database directly. It's an interface — you plug in a driver for PostgreSQL, MySQL, or whatever you're using. It's perfectly fine for small and medium projects.

## Visibility

There are no `public` or `private` keywords. The rule is the first letter:

- **Uppercase identifier** → exported (visible outside the package).
- **Lowercase identifier** → private to the package.

```go
func PublicHandler() {}   // exported
func parseBody() {}       // package-private
```

That's it. The whole access control model.

## Variables and the zero value

Go has no `null` for most types. Every type has a **zero value** that it gets automatically:

| Type | Zero value |
| --- | --- |
| Numbers | `0` |
| `string` | `""` |
| `bool` | `false` |
| Pointers, slices, maps, channels, functions, interfaces, `error` | `nil` |

So `nil` exists, but only for those reference-like types. An `int` is never null. A `struct` is never null — it's a struct with all its fields zeroed.

This kills a whole category of bugs and creates a smaller one: an uninitialized map or slice is `nil`, and writing to a `nil` map panics.

```go
var m map[string]int
m["a"] = 1        // panic: assignment to entry in nil map

m = make(map[string]int)
m["a"] = 1        // fine
```

## `make` vs `new`

- **`make`** builds things that need internal structure to work. Only three types: `map`, `slice`, `channel`.
- **`new`** just allocates memory and returns a pointer to the zero value.

In practice you use `make` for collections and `&Struct{}` for pointers to structs. `new` is rare.

## Loops

`for` is the only loop in Go. There is no `while`, no `do-while`.

```go
for i := 0; i < 10; i++ {}     // classic
for condition {}                // this is your while
for {}                          // infinite, exit with break
for i, v := range slice {}      // iterate
```

To iterate a slice or a map you use `range`. Worth remembering: **map iteration order is random by design.** Go deliberately randomizes it so you never write code that depends on it.

## Functions

The shape is `func` + name + parameters + return type:

```go
func Divide(a, b float64) (float64, error) {
    if b == 0 {
        return 0, errors.New("division by zero")
    }
    return a / b, nil
}
```

Functions can return multiple values at once, which is how the error convention works in the first place.

## Methods and receivers

You can attach a function to a type using a **receiver** — the part in parentheses before the function name:

```go
func (u User) FullName() string {
    return u.Name + " " + u.LastName
}
```

Mentally, that's equivalent to:

```go
func FullName(u User) string { ... }
```

The receiver is just a first argument with nicer syntax. Two flavors:

- **Value receiver** `func (u User)` — gets a copy. Cannot mutate the original.
- **Pointer receiver** `func (u *User)` — gets the address. Can mutate the original.

Rule of thumb: if the method changes state, or the struct is large, use a pointer receiver. And keep it consistent — don't mix both on the same type.

## Structs

A struct groups related data. It's similar to a Java class, but it only holds **data**. Methods are defined separately and associated through the receiver. In Java, the class holds both data and behavior; in Go they're split.

```go
type User struct {
    ID    string
    Name  string
    Email string
}

u := User{ID: "1", Name: "Ana"}
```

## Interfaces are implicit

This is one of the real differences from Java. There is no `implements`. A type satisfies an interface **automatically** if it has all the required methods.

```go
type Notifier interface {
    Send(msg string) error
}

type EmailSender struct{}

func (e EmailSender) Send(msg string) error { return nil }
// EmailSender now satisfies Notifier. Nothing was declared.
```

An interface defines the **contract** — the "what". A struct provides the "how".

The practical consequence: you can define an interface in the package that *consumes* it, not the one that provides it. Small interfaces, defined where they're used. `io.Reader` has one method and half the standard library speaks it.

## Slices, arrays and maps

- **Array**: fixed-size block of memory holding values directly. Its length is part of its type and never changes.
- **Slice**: a structure pointing at an internal array, tracking how many elements it uses (`len`) and how much room it has (`cap`). Grows with `append`. This is the Java `ArrayList`.
- **Map**: key → value dictionary, with the comma-ok pattern for lookups.

```go
nums := []int{1, 2, 3}
nums = append(nums, 4)

ages := map[string]int{"ana": 30}
age, ok := ages["ana"]   // comma-ok: ok is false if the key is missing
```

The comma-ok pattern matters: `ages["missing"]` returns `0`, not an error. Without `ok` you cannot tell "the value is zero" from "the key doesn't exist".

## Pointers

Go passes arguments **by value**, always. If you pass a normal type, the function gets a copy and changes don't affect the original. If you pass a pointer (`*Type`), the function gets the address and can modify the original through `*`. You take the address with `&`.

```go
func rename(u *User, name string) {
    u.Name = name          // modifies the original
}

rename(&user, "Ana")
```

## `defer`

`defer` schedules a function call to run right before the surrounding function returns. Deferred calls execute **in reverse order** — last in, first out.

```go
func readFile(path string) error {
    f, err := os.Open(path)
    if err != nil {
        return err
    }
    defer f.Close()        // runs no matter how we return
    // ...
}
```

Use it for releasing resources: closing files and database connections, unlocking mutexes.

## Enums with `iota`

Go has no native enums. You build them with a type, constants, and `iota` — which assigns consecutive integers starting at 0.

```go
type Status int

const (
    Pending  Status = iota // 0
    Approved               // 1
    Rejected               // 2
)

status := Pending
```

`const` because these values must never change. You use `:=` afterwards for the variable that holds one of them.

The point of an enum is avoiding magic numbers and magic strings. `Pending` tells you something; `0` doesn't.

## JSON

```go
b, err := json.Marshal(user)      // Go → JSON
err = json.Unmarshal(b, &user)    // JSON → Go
```

Struct tags control the field names:

```go
type User struct {
    ID    string `json:"id"`
    Email string `json:"email,omitempty"`
}
```

Remember the visibility rule — **only exported fields get marshaled**. A lowercase field is invisible to `encoding/json`, and this is a classic silent bug.

## Models, interfaces and DTOs

- **Struct (model)**: stores data.
- **Interface**: defines a method contract.
- **DTO** (Data Transfer Object): the structure representing what goes in and out of your API.

A struct can implement many interfaces at once, as long as it has the methods.

# Part 2 — Advanced

## Composition over inheritance

Go has **no inheritance**. You reuse code by embedding one struct inside another.

- Inheritance: a Car **IS A** Vehicle.
- Composition: a Car **HAS AN** Engine.

Instead of a `Dog` inheriting from `Animal`, you embed `Animal` into `Dog` and get its fields and methods promoted:

```go
type Animal struct{ Name string }

func (a Animal) Speak() string { return a.Name + " makes a sound" }

type Dog struct {
    Animal          // embedded — no field name
    Breed string
}

d := Dog{Animal: Animal{Name: "Rex"}, Breed: "Lab"}
d.Speak()           // works — promoted from Animal
```

This favors flat composition over deep class hierarchies, which is easier to test and easier to change.

## Goroutines

A goroutine is a function running concurrently, managed by the Go runtime instead of the OS. They start at a couple of kilobytes of stack and grow on demand, which is why thousands of them are fine.

```go
go doWork()
```

The catch: `main` does not wait for them. If `main` returns, everything dies.

## Channels

A channel is a **typed pipe** for communicating between goroutines.

```go
ch := make(chan int)      // unbuffered
ch := make(chan int, 10)  // buffered, capacity 10
```

**Unbuffered** (`make(chan T)`): the send blocks until another goroutine receives. It synchronizes 1-to-1 — a handoff, a meeting point. Use it when you need to be sure the receiver actually got the value before moving on.

**Buffered** (`make(chan T, N)`): accepts up to N elements without blocking. The send only blocks when the buffer is full; the receive blocks when it's empty. Use it when you want to decouple the two sides and let the producer run a bit ahead of the consumer.

Channels also close, and a closed channel can still be drained:

```go
close(ch)
v, ok := <-ch    // ok is false once the channel is closed and empty
```

Rule: **the sender closes the channel, never the receiver.** Sending on a closed channel panics.

## `select`

`select` waits on several channels at once and runs the case of whichever is ready first. It fires **once** — to keep listening you wrap it in a `for`.

```go
for {
    select {
    case msg := <-messages:
        handle(msg)
    case <-ctx.Done():
        return ctx.Err()
    case <-time.After(5 * time.Second):
        return errors.New("timeout")
    }
}
```

That `ctx.Done()` case is the standard way to make a worker cancellable.

## Concurrency patterns

Four names that come up constantly, in plain language:

- **Producer–Consumer** — someone creates work, someone else does it.
- **Worker Pool** — a fixed team of workers pulling from one queue.
- **Fan-Out** — one queue distributing work to several workers.
- **Fan-In** — several sources loading work into the same queue.

A worker pool in practice:

```go
jobs := make(chan Job, 100)
results := make(chan Result, 100)

for w := 0; w < 5; w++ {          // 5 workers
    go func() {
        for j := range jobs {      // range over a channel until it closes
            results <- process(j)
        }
    }()
}
```

## Handling errors inside goroutines

A goroutine cannot return a value to its caller. The usual approach is to send errors into a channel that one goroutine owns and handles. Alternatively, wait for the goroutines to finish and collect what failed.

```go
errs := make(chan error, len(jobs))
// each worker: errs <- doWork()
```

For anything non-trivial, `errgroup` from `golang.org/x/sync` does this properly: it runs a group of goroutines, returns the first error, and cancels the shared context.

## `sync.WaitGroup`

Waits for a group of goroutines to finish before continuing. You have to tell it **how many** you're waiting for.

```go
var wg sync.WaitGroup

for _, url := range urls {
    wg.Add(1)
    go func(u string) {
        defer wg.Done()
        fetch(u)
    }(url)
}

wg.Wait()      // blocks until all Done() calls land
```

`defer wg.Done()` is the safe form — it runs even if the goroutine panics or returns early.

## Race conditions

A race condition is when the result depends on the order in which goroutines touch shared state.

- Shared variable or state → `sync.Mutex`.
- Shared map → `sync.Map`, or a normal map guarded by a mutex.
- Mostly reads, few writes → `sync.RWMutex` (many readers or one writer).

The native map is **not safe for concurrent writes** — it doesn't corrupt silently, it crashes with a `fatal error`. That's deliberate.

To detect races, use the race detector:

```bash
go run -race main.go
go test -race ./...
```

Run your test suite with `-race` in CI. It finds things code review never will.

## `context.Context`

`context` propagates cancellation, deadlines and request-scoped values through the layers of an operation. It's fundamental in microservices.

A function takes a context so it can react to a cancellation or a timeout, or pass it down to a dependency.

```go
type Context interface {
    Deadline() (deadline time.Time, ok bool)
    Done() <-chan struct{}
    Err() error
    Value(key any) any
}
```

```go
ctx, cancel := context.WithTimeout(context.Background(), 3*time.Second)
defer cancel()

rows, err := db.QueryContext(ctx, "SELECT ...")
```

`context.WithValue()` stores small request-scoped data — a userID, requestID, traceID. Small and specific, never large objects and never as a way to avoid passing parameters.

Two conventions: context is always the **first parameter**, named `ctx`, and you always call `cancel()` — usually with `defer` — to release the resources.

## Deadlocks

A deadlock is goroutines waiting for something that will never happen — like a Promise that never resolves. It's not a feature, it's an error state caused by bad concurrency design.

The classic case is an unbuffered channel with nobody on the other end:

```go
ch := make(chan int)
ch <- 1     // fatal error: all goroutines are asleep - deadlock!
```

Go detects this specific case at runtime when *every* goroutine is blocked. It cannot detect the partial cases, which are the ones that hurt in production.

## Garbage collector, stack and heap

Go manages memory automatically with a garbage collector. The compiler uses **escape analysis** to decide whether a variable lives on the stack or escapes to the heap:

- Variables that don't outlive the function → **stack**, freed for free on return.
- Variables referenced outside their scope → **heap**, cleaned up by the GC when nothing references them.

Day to day you don't control this manually. It's worth understanding to reason about the cost of allocations, avoid generating unnecessary garbage, and debug memory and performance problems.

You can see the compiler's decisions:

```bash
go build -gcflags="-m" ./...
```

## `panic` and `recover`

- **`panic`** raises a fatal, unexpected error that interrupts normal execution and unwinds the stack.
- **`recover`** catches a panic — only from inside a `defer` — so the application can keep running.

The standard use is API middleware: catch any unexpected panic during a request, log it, return a 500, and keep the server alive instead of taking down every other in-flight request.

```go
func Recovery(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        defer func() {
            if err := recover(); err != nil {
                log.Printf("panic: %v", err)
                http.Error(w, "internal error", http.StatusInternalServerError)
            }
        }()
        next.ServeHTTP(w, r)
    })
}
```

`panic`/`recover` are the closest thing to `throw`/`catch`, but they do **not** replace error handling with `error`. They're reserved for critical or genuinely unexpected situations. Expected errors are returned as values.

## Wrapping errors

`errors` is small but the semantics matter:

```go
if err != nil {
    return fmt.Errorf("fetching user %s: %w", id, err)   // %w wraps
}
```

- `errors.Is(err, ErrNotFound)` — is this error, anywhere in the chain, that sentinel?
- `errors.As(err, &myErr)` — is there an error of this type in the chain, and give it to me.

`%w` preserves the chain. `%v` flattens it to a string and you lose the ability to inspect it.

## Generics

Generics let you write a single function or type that works with different data types, instead of duplicating identical logic that only differs by type.

```go
func Sum[T int | float64](a, b T) T {
    return a + b
}

func Map[T, U any](in []T, f func(T) U) []U {
    out := make([]U, 0, len(in))
    for _, v := range in {
        out = append(out, f(v))
    }
    return out
}
```

Use them when the logic is genuinely identical across types. If you find yourself fighting the constraints, an interface was probably the right answer.

## `defer` and named returns

A `defer` runs just before the function ends. If the function uses **named return values**, the deferred call can modify what gets returned, because it's changing the actual variable — while a normal `return` returns a copy of the value.

```go
func do() (err error) {
    defer func() {
        if err != nil {
            err = fmt.Errorf("do failed: %w", err)
        }
    }()
    return doSomething()
}
```

This exists and it's occasionally the cleanest option, but it's usually avoided because it makes the code harder to follow.

## Project structure

A large Go application is usually organized **by domain** — users, messages, groups, payments — rather than by technical layer. Each domain has its own handlers, services and repositories. The services hold the business logic and coordinate components like databases, queues, websockets and external services.

```
/internal
  /user
    handler.go
    service.go
    repository.go
  /payment
    handler.go
    service.go
    repository.go
```

Anything under `internal/` cannot be imported from outside the module. It's compiler-enforced encapsulation, and it's free.

## Event bus

An event bus lets different parts of an application communicate without knowing about each other. Instead of one component calling another directly, it **publishes an event**, and any interested component receives it.

The trade-off is real: you gain decoupling, you lose the ability to follow the flow by reading the code. Worth it when you have several unrelated reactions to the same fact ("user registered" → send email, create wallet, log analytics). Not worth it for a call between two components that will always be exactly two.

## The short version

If you remember five things from this:

1. **Errors are values.** Handle them where they happen.
2. **Interfaces are implicit and should be small.** Define them where you consume them.
3. **Unbuffered channels synchronize; buffered ones decouple.** Pick on purpose.
4. **Always pass `ctx` first**, and always `defer cancel()`.
5. **Run your tests with `-race`.**
