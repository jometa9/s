## Technical Interviews — Walking Through My Approach, Trade-offs, Reasoning Out Loud

## Introduction

Step 13 — the last block of the plan from *"How I'm training myself to stop sounding basic in english meetings"*.

I already wrote about the behavioral round in *"Behavioral Interview — How I Present My Background"*. This one is the technical round: a problem, a shared editor, and someone watching me think.

It's the hardest block, because everything from the previous twelve is being evaluated at once.

---

## Why This Matters

In a technical interview, **silence is the failure mode**. Not a wrong answer — silence.

If I solve the problem without speaking, the interviewer learns nothing about how I think. If I talk through a *worse* solution clearly, explain the trade-off, and improve it out loud, they learn everything they wanted to know.

Which is brutal for a non-native speaker, because the exact thing I do under pressure — go quiet and think in Spanish — is the one behaviour that gets scored lowest.

So the skill isn't the algorithm. It's narrating while my hands are busy.

---

## The Core Structure

Four phases. Say each one out loud as you enter it, so the interviewer always knows where you are:

```
1. Restate    "So the input is X and I need to return Y —
               is that right?"
2. Approach   "My first idea is a hash map. O(n) time, O(n)
               space. Should I go with that?"
3. Code       "I'm handling the empty case first..."
4. Review     "Let me trace through an example. What about
               duplicates?"
```

Announcing the phase buys you thinking time *and* makes you look organized. Two problems, one sentence.

## Situations & Examples

### Situation 1: You Just Heard the Problem

**Context:** First thirty seconds. The instinct is to start typing immediately.

**Bad way:**
```
[Silence. Start coding.]
```
❌ You might be solving the wrong problem, and they can't see any of your thinking.

**Good way:**
```
Let me make sure I've got it: I get a list of transactions and
I need to find the first one that repeats. If nothing repeats,
what should I return — null?
```
✅ This is **Lesson 2 (asking for clarification)** in the highest-stakes place it appears.

Always ask about the empty case and about invalid input. Interviewers are waiting for that question.

---

### Situation 2: Talking Before You Have the Solution

**Context:** You don't know the answer yet. This is normal and you still have to talk.

**Bad way:**
```
[Two minutes of silence while thinking]
```
❌ Two minutes feels like ten to the person watching.

**Good way:**
```
Let me think out loud. The brute force is comparing every pair,
which is O(n²) — that works but it's slow. I feel like there's
a way to do it in one pass if I keep track of what I've seen...
yeah, a set would do that.
```
✅ Narrating a wrong idea and rejecting it is *worth points*. It's exactly what they're measuring.

**"Let me think out loud"** is the single most useful phrase in this block.

---

### Situation 3: Explaining a Trade-off

**Context:** Two valid approaches. How you choose matters more than which you choose.

**Bad way:**
```
I'll use a hash map.
```
❌ Correct, and reveals nothing.

**Good way:**
```
Two options. Sorting first is O(n log n) with no extra memory.
A hash set is O(n) but uses O(n) space. I'd go with the hash
set since we're not told memory is tight — but if this ran on
a device, I'd sort instead.
```
✅ Both options, both costs, a choice with a reason, and the condition that would flip it.

This is **Lesson 4 (giving your opinion)** with a stronger requirement: always give the reason.

---

### Situation 4: You're Stuck

**Context:** You've stopped making progress and you can feel it.

**Bad way:**
```
[Keep staring silently, panicking]
```
❌ The silence hurts more than being stuck.

**Good way:**
```
I'm a bit stuck here. What I have works for the normal case,
but I can't see how to handle nested groups without recursion,
and I'm not sure recursion is right. Can I talk through where
I am?
```
✅ Being stuck is fine. Being stuck *silently* is what fails.

Most interviewers will give you a hint. They can't if they don't know where you are.

---

### Situation 5: You Wrote a Bug

**Context:** You spot a mistake in your own code.

**Bad way:**
```
[Quietly fix it, hope they didn't notice]
```
❌ They noticed. And catching your own bug is a positive signal you just threw away.

**Good way:**
```
Actually, wait — this breaks if the list is empty. `arr[0]`
throws before we get to the loop. Let me add a guard at the top.
```
✅ Catching your own bug out loud is one of the strongest moments in an interview.

---

### Situation 6: The Interviewer Suggests Something

**Context:** They ask "have you considered X?" That's a hint, not a question.

**Bad way:**
```
[Defend your approach]
```
❌ You're arguing with someone trying to help you pass.

**Good way:**
```
Good point — I hadn't thought about that. If I sort first, then
the duplicates are adjacent and I don't need the set at all.
Let me redo it that way.
```
✅ Take the hint, say what changed, adjust. **Lesson 5**, in an interview.

---

### Situation 7: You Don't Know Something

**Context:** They ask about a technology you haven't used.

**Bad way:**
```
Yeah, I've worked with that.
```
❌ The follow-up question will expose it in ten seconds.

**Good way:**
```
I haven't used Kafka in production — I've read about it and I
understand the model, but I'd be learning on the job. I've done
something similar with SQS, where the ordering guarantees are
different.
```
✅ Honest, then pivot to the closest thing you *have* done. That's a strong answer, not a weak one.

---

### Situation 8: Your English Fails Mid-Sentence

**Context:** You lose a word in the middle of explaining. It will happen.

**Bad way:**
```
[Freeze. Panic. Apologize.]
```
❌ Turns a two-second gap into a visible collapse.

**Good way:**
```
Sorry, I've lost the word — the thing where a function calls
itself. Recursion, right. So with recursion we'd...
```
✅ Name the gap, describe around it, keep going. Nobody is scoring your vocabulary.

**Lesson 3 (buying time)** matters more here than anywhere else.

## Phrases

### Clarifying the problem

```
Let me make sure I understand the problem.
So the input is X and I return Y — is that right?
What should I return for an empty input?
Can I assume the list is sorted?
Are there constraints on size or memory?
Should I handle invalid input, or assume it's clean?
```

### Thinking out loud

```
Let me think out loud for a second.
My first instinct is X, but I think there's a better way.
Brute force would be O(n²) — let me see if I can do better.
Let me start with something that works and optimize after.
I'm going to try X and see where it breaks.
```

### Trade-offs

```
There are two approaches here.
The trade-off is time versus memory.
That's O(n log n) but no extra space.
I'd go with X because... but if Y mattered more, I'd switch.
This is the simpler one, which matters if it's read often.
```

### While coding

```
I'm handling the empty case first.
I'll come back to error handling in a second.
Let me name this properly — it'll matter when I read it back.
I'm going to extract this into a helper for readability.
```

### Being stuck

```
I'm a bit stuck. Let me talk through where I am.
I can see the problem but not the solution yet.
Can I have a minute to think about this?
Am I on the right track here?
Would a hint be okay?
```

### Catching your own bug

```
Actually, wait — this breaks if...
Let me trace through an example... yeah, that's wrong.
I need a guard here.
Good, that would have failed on duplicates.
```

### Reviewing at the end

```
Let me test it with an example.
Edge cases: empty list, one element, all duplicates.
Time complexity is O(n), space is O(n).
If I had more time I'd add tests for X.
Is there anything you'd want me to improve?
```

### When you don't know

```
I haven't used that in production, but I understand the idea.
I've done something similar with X.
I don't know that one — how does it compare to Y?
I'd have to look that up. My guess would be X, but I'd verify.
```

### When English fails

```
Sorry, I've lost the word — the thing that...
How do I say this... basically, it...
Let me rephrase that.
Give me a second to find the right words.
```

## Key Rules

1. **Silence is the only real failure**
   - A worse solution explained clearly beats a perfect one in silence.
2. **Restate the problem before you code**
   - Costs twenty seconds, catches misunderstandings that cost the whole interview.
3. **Announce the phase you're in**
   - "Let me think about the approach first" tells them where you are and buys you time.
4. **Say the trade-off, always**
   - Choosing without a reason looks like guessing.
5. **Being stuck out loud is fine**
   - Say where you are. Most interviewers will help.
6. **Take every hint immediately**
   - "Have you considered X?" is not a question.
7. **Don't fake experience**
   - "I haven't used it, but I've done something similar" is a strong answer.
8. **Practise out loud, in English, alone**
   - Solving problems while narrating in English is the actual rep. Silent practice trains the wrong thing.

---

## What NOT to Do

| ❌ Don't | ✅ Do |
|---|---|
| Start coding immediately | "Let me make sure I understand the problem." |
| Think silently for two minutes | "Let me think out loud — brute force would be..." |
| "I'll use a hash map." | "Hash map is O(n) time, O(n) space — the trade-off is..." |
| Freeze when stuck | "I'm stuck. Let me talk through where I am." |
| Fix your bug quietly | "Wait — this breaks on empty input." |
| Argue with a hint | "Good point, I hadn't considered that." |
| Claim experience you don't have | "I haven't used it, but I've done something similar." |
| Apologize for your English | Describe around the missing word and continue. |

## The Bigger Picture

That's all thirteen blocks.

Writing them out, the thing that surprised me is how little of this was ever about English. Almost every block came down to the same handful of moves: say the thing out loud, say it early, give the reason, and don't disappear when it gets uncomfortable.

The phrases are just the delivery mechanism. What was actually stopping me was the belief that I needed better English before I was allowed to speak.

The interview block makes that obvious, because it's the one place where staying quiet is explicitly scored as failure. But it's true in every meeting — nobody can tell the difference between a developer who has nothing to say and one who can't get it out fast enough. From the outside those look identical.

So now I have the list. Thirteen blocks, a few hundred phrases, and one rule that covers all of it: **out loud, on time, with a reason.**

Now I have to actually run it. I'll report back.
