# Part 1 — The basics

## Fundamentals

SQL is a **declarative** language: you describe *what* you want, not *how* to get it. The engine decides the execution plan. This is why two queries that return the same rows can differ by three orders of magnitude in speed — you're negotiating with an optimizer, not writing instructions.

The four families of commands:

| Family | Commands | Purpose |
| --- | --- | --- |
| **DDL** | `CREATE`, `ALTER`, `DROP`, `TRUNCATE` | Define structure |
| **DML** | `SELECT`, `INSERT`, `UPDATE`, `DELETE` | Manipulate data |
| **DCL** | `GRANT`, `REVOKE` | Permissions |
| **TCL** | `BEGIN`, `COMMIT`, `ROLLBACK` | Transactions |

## The vocabulary

- **Database** — an organized set of information.
- **Table** — a set of rows and columns.
- **Row** (record) — one entry in the table.
- **Column** (field) — one attribute of the records.
- **Primary Key (PK)** — uniquely identifies each row.
- **Foreign Key (FK)** — relates one table to another.
- **Index** — speeds up lookups.
- **Constraint** — a rule defined in the database that guarantees data integrity: unique keys, valid relationships, required values.
- **View** — a saved query that behaves like a virtual table.
- **Cardinality** — how many records on each side of a relationship (1:1, 1:N, N:M).

## Common types

`INT`/`BIGINT`, `DECIMAL`/`NUMERIC`, `VARCHAR`/`TEXT`, `BOOLEAN`, `DATE`/`TIMESTAMP`, `UUID`.

**For money, always `DECIMAL`, never `FLOAT` or `DOUBLE.`** Floating point cannot represent `0.1` exactly, so your balances will drift by fractions of a cent and eventually someone will notice. This is a real interview question and a real production bug.

## NULL is not a value

`NULL` means **unknown**, not "empty" and not "zero".

The consequence catches everyone: `NULL = NULL` evaluates to `NULL`, not `true`. You compare with `IS NULL` / `IS NOT NULL`.

```sql
SELECT * FROM users WHERE deleted_at IS NULL;   -- correct
SELECT * FROM users WHERE deleted_at = NULL;    -- returns nothing, ever
```

It propagates too: `5 + NULL` is `NULL`, and `COUNT(column)` skips NULLs while `COUNT(*)` doesn't.

`COALESCE(a, b, c)` returns the first non-null value, which is how you give NULL a default in a result set.

## The basic operations

```sql
SELECT name, salary
FROM employees
WHERE salary > 50000
ORDER BY salary DESC
LIMIT 10;
```

- **`SELECT`** retrieves data. **`INSERT`** adds records. **`UPDATE`** modifies them. **`DELETE`** removes them.
- **`WHERE`** filters rows. **`ORDER BY`** sorts. **`LIMIT`** caps the row count, **`OFFSET`** skips rows.
- **`DISTINCT`** removes duplicate rows from the result.
- **`AS`** assigns an alias.
- **`LIKE`** matches patterns, **`IN`** checks membership in a list, **`BETWEEN`** filters a range.
- **`CASE`** gives you conditional logic inside a query.

## The logical execution order

This is the single most useful thing to memorize, because it explains half of SQL's error messages.

You *write* the query in one order. The engine *evaluates* it in another:

```
FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT
```

Two consequences that come up constantly:

- **`WHERE` filters rows before grouping; `HAVING` filters groups after.**
- **You cannot use a `SELECT` alias inside `WHERE`**, because `WHERE` runs before `SELECT` exists. You *can* use it in `ORDER BY`, which runs after.

```sql
SELECT salary * 12 AS annual
FROM employees
WHERE annual > 100000     -- error: "annual" doesn't exist yet
ORDER BY annual;          -- fine: ORDER BY runs after SELECT
```

## JOINs

The most-asked topic in SQL interviews, without much competition.

| JOIN | Returns |
| --- | --- |
| `INNER JOIN` | Only rows with a match in both tables (A ∩ B) |
| `LEFT JOIN` | All rows from the left; NULL where there's no match on the right |
| `RIGHT JOIN` | All rows from the right; NULL where there's no match on the left |
| `FULL OUTER JOIN` | All rows from both; NULL wherever a match is missing |
| `CROSS JOIN` | Cartesian product — every combination |
| `SELF JOIN` | A table against itself (employee → their manager in the same table) |

```sql
SELECT e.name, d.name AS dept
FROM employees e
INNER JOIN departments d ON e.dept_id = d.id;
```

**Anti-join** — finding the rows that have *no* match — is a classic exercise: `LEFT JOIN` plus `WHERE b.id IS NULL`.

```sql
-- customers who have never placed an order
SELECT c.*
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id
WHERE o.id IS NULL;
```

**The trap worth knowing.** `ON` defines how the tables are matched; `WHERE` filters afterwards. A filter on the right-hand table placed in the `WHERE` clause silently turns your `LEFT JOIN` into an `INNER JOIN`, because the NULL rows you were trying to keep get filtered out:

```sql
-- broken: drops customers with no orders
LEFT JOIN orders o ON o.customer_id = c.id
WHERE o.status = 'paid'

-- correct: the condition belongs in the ON
LEFT JOIN orders o ON o.customer_id = c.id AND o.status = 'paid'
```

## Aggregation and GROUP BY

Aggregate functions: `COUNT`, `SUM`, `AVG`, `MIN`, `MAX`.

- `COUNT(*)` counts rows.
- `COUNT(column)` ignores NULLs.
- `COUNT(DISTINCT col)` counts distinct values.

`GROUP BY` groups rows sharing a value so aggregates apply per group. **Every non-aggregated column in the `SELECT` must appear in the `GROUP BY`.**

```sql
SELECT dept_id, COUNT(*) AS total, AVG(salary) AS average
FROM employees
GROUP BY dept_id
HAVING COUNT(*) > 5;
```

`WHERE` filters rows before aggregating; `HAVING` filters groups after. If you can express a filter in `WHERE`, do — it reduces the rows before the expensive part.

## UNION and set operations

- **`UNION`** combines results and removes duplicates.
- **`UNION ALL`** combines results and keeps duplicates.

`UNION ALL` is faster because it skips the deduplication pass. If you know there are no duplicates, or you don't care, use it.

## INSERT / UPDATE / DELETE

```sql
INSERT INTO users (name, email) VALUES ('Ana', 'ana@x.com');
UPDATE users SET balance = balance + 100 WHERE id = 1;
DELETE FROM users WHERE id = 1;
```

**Always write the `WHERE` in `UPDATE` and `DELETE`.** Without it you hit the entire table. The habit that saves you: write the `WHERE` first, run it as a `SELECT`, confirm the row count, then change `SELECT` to `UPDATE`.

**UPSERT** — insert or update if it already exists:

```sql
-- PostgreSQL
INSERT INTO users (id, name) VALUES (1, 'Ana')
ON CONFLICT (id) DO UPDATE SET name = EXCLUDED.name;

-- MySQL
INSERT INTO users (id, name) VALUES (1, 'Ana')
ON DUPLICATE KEY UPDATE name = VALUES(name);
```

**`TRUNCATE`** empties a table fast — no condition, no row-by-row logging. `DELETE` is transactional and filterable. Different tools for different jobs.

## Constraints

- **`PRIMARY KEY`** — identifies the row (unique + not null).
- **`FOREIGN KEY`** — references another table and guarantees referential integrity: you cannot point at something that doesn't exist. Options: `ON DELETE CASCADE`, `SET NULL`, `RESTRICT`.
- **`UNIQUE`** — no duplicate values.
- **`NOT NULL`** — a value is required.
- **`CHECK`** — validates a condition, e.g. `CHECK (balance >= 0)`.
- **`DEFAULT`** — a value used when none is supplied.

Constraints are the cheapest correctness guarantee you will ever get. A `CHECK (balance >= 0)` in the database is enforced no matter which service, script or intern writes to that table. The same rule in application code is enforced only where somebody remembered to write it.

# Part 2 — Advanced

## Subqueries and CTEs

A **subquery** is a query inside another query. It can appear in `SELECT`, `FROM` or `WHERE`.

A **correlated subquery** references the outer row, so it runs once per row — much slower.

`IN` / `EXISTS` / `NOT EXISTS` check membership. `EXISTS` is usually more efficient with large subqueries because it stops at the first match instead of building the full result set.

A **CTE** (Common Table Expression) with `WITH` names a subquery for readability and reuse:

```sql
WITH high_sales AS (
    SELECT seller_id, SUM(amount) AS total
    FROM sales
    GROUP BY seller_id
    HAVING SUM(amount) > 10000
)
SELECT s.name, hs.total
FROM high_sales hs
JOIN sellers s ON s.id = hs.seller_id;
```

CTEs can be **recursive**, which is how you query hierarchies and trees — an org chart, a category tree, a thread of replies:

```sql
WITH RECURSIVE subordinates AS (
    SELECT id, name, manager_id FROM employees WHERE id = 1
    UNION ALL
    SELECT e.id, e.name, e.manager_id
    FROM employees e
    JOIN subordinates s ON e.manager_id = s.id
)
SELECT * FROM subordinates;
```

One caveat: a CTE improves readability, and in some engines it's an optimization fence — PostgreSQL materialized every CTE until version 12, which occasionally made queries *slower*. Readability first, but check the plan.

## Window functions

This is the topic that separates "knows SQL" from "knows SQL". Window functions operate over a set of related rows **without collapsing the result**, which is exactly what `GROUP BY` cannot do.

`OVER (PARTITION BY ... ORDER BY ...)` defines the window.

**Ranking:**
- `ROW_NUMBER()` — unique sequential number.
- `RANK()` — ties share a rank and leave gaps (1, 2, 2, 4).
- `DENSE_RANK()` — ties share a rank with no gaps (1, 2, 2, 3).

**Others:** `LAG()` / `LEAD()` (previous/next row), `SUM() OVER` (running total), `NTILE()` (buckets).

```sql
-- top salary per department
SELECT * FROM (
  SELECT name, dept_id, salary,
         ROW_NUMBER() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rn
  FROM employees
) t
WHERE rn = 1;
```

```sql
-- month-over-month change
SELECT month, revenue,
       revenue - LAG(revenue) OVER (ORDER BY month) AS delta
FROM monthly_revenue;
```

You need the subquery in the first example because window functions are computed at `SELECT` time — after `WHERE` — so you can't filter on `rn` in the same query level.

## Indexes

An index is a data structure — normally a **B-tree** — that speeds up lookups, at the cost of disk space and slower writes.

The analogy: the index at the back of a book. Without it you read every page (full table scan, O(n)). With it, you jump straight to the right page (O(log n)).

- **Speeds up**: `SELECT`, `WHERE`, `JOIN`, `ORDER BY` on the indexed column.
- **Slows down**: `INSERT`, `UPDATE`, `DELETE` — every write must maintain every index.

**The rule**: index the columns that actually appear in `WHERE`, `JOIN` and `ORDER BY`. Not every column. An unnecessary index is dead weight you pay for on every single write.

**Types:**
- **Primary key** — unique index, created automatically.
- **Unique** — enforces uniqueness and indexes.
- **Composite** — spans multiple columns.
- **Covering** — contains every column the query asks for, so the engine answers from the index and never touches the table.
- **Clustered** — physically orders the table data by the index. You get one per table.
- **Non-clustered** — a separate structure pointing at the rows.

```sql
CREATE INDEX idx_employees_dept ON employees(dept_id);
```

**The leftmost prefix rule.** A composite index `INDEX(a, b)` can serve queries filtering on `a`, or on `a AND b` — but **not** on `b` alone. Think of a phone book sorted by (last name, first name): you can find everyone named García, and García Ana specifically, but you cannot find everyone named Ana. Column order in a composite index is a decision, not a formality.

## When an index is not used

Knowing this is more valuable than knowing how to create one, because "I added an index and nothing changed" is a weekly occurrence.

**A function applied to the column** kills the index:

```sql
WHERE YEAR(created_at) = 2024                  -- no index
WHERE created_at >= '2024-01-01'
  AND created_at <  '2025-01-01'               -- uses the index
```

The index stores `created_at` values, not `YEAR(created_at)` values. Rewrite as a range.

**Leading wildcard in LIKE:**

```sql
WHERE name LIKE '%ana'    -- no index
WHERE name LIKE 'ana%'    -- uses the index
```

Same reason: a B-tree is sorted by prefix. For real text search you want a full-text index or a trigram index, not `LIKE`.

**Low cardinality columns** — a boolean, a status with three values. The index barely helps because each entry points at a huge fraction of the table.

**When the query returns most of the table** — the engine correctly decides a sequential scan is cheaper than jumping through an index and back to the heap thousands of times.

**Implicit type conversion** — comparing a `VARCHAR` column against a number forces a cast on the column and drops the index. This one is silent and painful.

## EXPLAIN and performance

`EXPLAIN` (or `EXPLAIN ANALYZE`, which actually runs the query) shows the execution plan: whether it uses an index or does a full scan, the join order, and the estimated row counts.

What to look for:

- **"Seq Scan" / "full table scan" on a large table** → warning sign.
- **"Index Scan"** → good.
- **Estimated rows wildly different from actual rows** → your statistics are stale; run `ANALYZE`. The optimizer makes bad choices when it has bad numbers.
- **Nested Loop over a large outer set** → often the shape of an N+1 problem.

**The N+1 problem**: one query returns N rows, and then you run another query for each one. 1 + N round trips instead of 1. It's the single most common performance bug in ORM-backed applications, and the fix is a `JOIN` or an eager fetch.

**`SELECT *`** pulls columns you don't need, costs extra I/O, and breaks covering indexes. Ask for what you use.

**Pagination**: `LIMIT`/`OFFSET` is fine for the first few pages. For large tables, `OFFSET 100000` forces the engine to read and discard 100,000 rows every time. **Keyset pagination** is much better:

```sql
-- instead of: LIMIT 20 OFFSET 100000
SELECT * FROM orders
WHERE id > :last_seen_id
ORDER BY id
LIMIT 20;
```

Constant time regardless of how deep you are. The trade-off is you lose the ability to jump to an arbitrary page number.

## Transactions and ACID

A **transaction** is a set of operations that execute as one unit: either all of them happen, or none of them do.

```sql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;  -- debit
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- credit
COMMIT;   -- if anything fails: ROLLBACK, and nothing is applied
```

The classic example is exactly this: a transfer cannot debit one account without crediting the other. There is no acceptable intermediate state.

**ACID:**

- **Atomicity** — all or nothing. If one step fails, everything rolls back.
- **Consistency** — the database moves from one valid state to another, respecting all constraints.
- **Isolation** — concurrent transactions don't interfere with each other, according to the isolation level.
- **Durability** — once committed, it survives a power cut, because it's persisted to disk.

## Isolation levels and anomalies

From least to most strict — more isolation means less concurrency:

| Level | Prevents |
| --- | --- |
| `READ UNCOMMITTED` | Nothing — allows dirty reads |
| `READ COMMITTED` | Dirty reads |
| `REPEATABLE READ` | Dirty + non-repeatable reads |
| `SERIALIZABLE` | Everything — as if transactions ran one after another |

**The anomalies:**

- **Dirty read** — reading data from another transaction that hasn't committed yet. It might roll back, and you acted on data that never existed.
- **Non-repeatable read** — you read the same row twice in one transaction and it changed in between.
- **Phantom read** — the same query returns new or missing *rows* between two reads.

Worth knowing the defaults: PostgreSQL and Oracle default to `READ COMMITTED`; MySQL's InnoDB defaults to `REPEATABLE READ`. If you assume the wrong one, your concurrency reasoning is wrong.

## Locking

**Optimistic locking** — assume no conflict, and verify at the end with a version column. If the version changed, someone else got there first and you retry.

```sql
UPDATE orders SET status = 'paid', version = version + 1
WHERE id = 1 AND version = 3;
-- 0 rows affected means someone else won; retry
```

**Pessimistic locking** — lock the row while you work with it:

```sql
BEGIN;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;   -- nobody else touches this row
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
COMMIT;
```

Optimistic for low-contention, read-heavy workloads. Pessimistic when conflicts are likely and retrying is expensive.

**Deadlock** — two transactions each waiting on a lock the other holds. The engine detects it and kills one of them. Prevention is the same as everywhere else: **always acquire locks in the same order**, and keep transactions short.

## Views, stored procedures and triggers

- **View** — a saved query exposed as a virtual table. Good for simplifying repeated joins and for controlling which columns a role can see. A **materialized view** stores the result and must be refreshed — fast reads, stale data.
- **Stored procedure** — a reusable block of SQL living in the database.
- **Trigger** — code that runs automatically on an event (insert, update, delete).

Both procedures and triggers are powerful and worth being careful with: logic in the database is invisible to anyone reading the application code, doesn't show up in your version control by default, and doesn't appear in a stack trace. Audit trails and integrity enforcement are reasonable uses. Business rules usually are not.

## Normalization

The process of organizing tables to avoid redundancy and update anomalies.

- **1NF** — atomic values. No lists inside a cell, no repeating groups.
- **2NF** — 1NF, plus every column depends on the *whole* key, not part of a composite key.
- **3NF** — 2NF, plus no transitive dependencies: a non-key column must not depend on another non-key column.

The mental shortcut: **every fact lives in exactly one place.**

**Denormalization** is duplicating data on purpose to speed up reads by avoiding joins, accepting that you now maintain consistency by hand. It's a legitimate, conscious trade-off — not a failure. Just make it deliberately, and after measuring.

## SQL vs NoSQL

- **SQL (relational)** — fixed schema, ACID transactions, joins, strong integrity. Ideal for structured data with real relationships.
- **NoSQL** — flexible schema, horizontal scaling, high volume. Families: document (MongoDB), key-value (Redis), columnar (Cassandra), graph (Neo4j).

**CAP theorem** — in a distributed system, when a network partition happens you choose between Consistency and Availability. You cannot have both during the partition. Classic SQL tends toward CP; many NoSQL systems toward AP.

It isn't "one is better". It depends on the case, and you'll often combine them — Postgres for the data, Redis for the cache. Also worth saying out loud in an interview: modern PostgreSQL has excellent JSON support, so "we need flexible schema" is no longer automatically an argument for leaving relational databases.

## Quick good practices

- **Name things consistently** — snake_case, and pick plural or singular but stay uniform.
- **Use transactions** for operations that must be atomic.
- **Parameterize your queries** (prepared statements) to prevent SQL injection. Never concatenate user input into SQL. Ever. This is the one item on this list that is a security vulnerability rather than a style preference.
- **Index according to real access patterns**, not "just in case".
- **Version your schema changes** with Flyway or Liquibase so migrations are reviewable and repeatable.
- **Keep transactions short.** A transaction held open across a network call to an external API is how you get lock pile-ups at 3am.

## The short version

1. **Learn the logical execution order.** It explains why your alias doesn't work in `WHERE` and why `HAVING` exists.
2. **A filter on the right table belongs in the `ON`**, not the `WHERE`, or your `LEFT JOIN` isn't one.
3. **A function on an indexed column disables the index.** Rewrite as a range.
4. **Window functions** are the highest-leverage thing to learn next if you already know joins.
5. **`DECIMAL` for money.** Always.
