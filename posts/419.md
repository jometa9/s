## Code Review — Feedback, Edge Cases, Asking Why

## Introduction

Step 9 of the plan from *"How I'm training myself to stop sounding basic in english meetings"*, and the first of the technical blocks.

Everything before this was spoken. Code review is mostly written — which sounds easier, and isn't. In a call I have tone, face, and the chance to fix a bad sentence immediately. In a PR comment I have a text box, and whatever I write is what people read three days later.

---

## Why This Matters

Written English strips out tone. A comment I meant as a question arrives as a complaint.

```
Why did you do it this way?
```

In my head that's genuine curiosity. On the screen it reads like an accusation. And I only find out when the reply comes back defensive.

The opposite problem is just as common. To avoid sounding harsh I write so softly that the actual issue disappears:

```
Maybe it's fine but I'm not sure, could be okay?
```

Which gets a 👍 and merged, bug included.

Code review is where the tone knob matters most, and where I have the most time to get it right — because nobody is waiting for me to answer in three seconds.

---

## The Core Structure

```
[Severity] + [Observation] + [Suggestion or question]

Non-blocking: if `items` comes back empty this throws on the
`[0]`. Would a guard clause work here, or is empty impossible
upstream?
```

- **Severity** — blocking or not. Say it first so they know how to read the rest.
- **Observation** — the fact, about the code.
- **Suggestion or question** — a way forward, or an honest question if you don't know.

Marking severity is the single biggest upgrade. Most tension in review comes from the author not knowing whether a comment is "you must fix this" or "just thinking out loud."

## Situations & Examples

### Situation 1: You Found a Real Bug

**Context:** This code will break. It has to be fixed.

**Bad way:**
```
This is wrong.
```
❌ True, useless, and it starts a defensive conversation.

**Good way:**
```
Blocking: if the user has no organization, `user.org.id` throws
here. I think that's possible for invited users who haven't
accepted yet. Optional chaining plus a fallback?
```
✅ Severity, the exact condition, when it happens, and a fix.

---

### Situation 2: You'd Do It Differently, But It's Not Wrong

**Context:** Style, preference, a pattern you like better. The code works.

**Bad way:**
```
I'd use a map here.
```
❌ No severity marker, so it reads as a change request. Now they're rewriting working code over your preference.

**Good way:**
```
Nit, ignore if you disagree: a map might read better than the
if-chain here. Totally fine as is.
```
✅ "Nit" is one of the most useful words in code review. It means: *this is optional*.

---

### Situation 3: You Don't Understand the Code

**Context:** You can't tell if it's clever or broken. This is the comment I most often *don't* write, because I'm scared of looking like I don't understand.

**Bad way:**
```
[Approve without understanding]
```
❌ You just approved something you can't review.

**Good way:**
```
Question, not a criticism — I'm not following why we need the
second loop. Is it handling duplicates? Might just be me
missing context.
```
✅ Labelling it as a question is what makes it safe to ask. And half the time, "I don't understand this" is itself the review.

---

### Situation 4: Asking About an Edge Case

**Context:** You suspect a case wasn't handled but you're not certain.

**Bad way:**
```
Did you test this?
```
❌ Reads as an accusation regardless of intent.

**Good way:**
```
What happens if the request times out halfway through? I might
be wrong about the retry behaviour here.
```
✅ Ask about the code, not about what they did or didn't do.

**"What happens if..."** is the best phrase in this whole block. It's neutral, it's specific, and if the case is handled the answer is just "it's handled."

---

### Situation 5: Approving

**Context:** It's good. Say something.

**Bad way:**
```
LGTM
```
❌ Fine, but it tells them nothing.

**Good way:**
```
LGTM. The error handling in the sync function is much clearer
than the old version — nice.
```
✅ Twelve extra words. Specific praise is rare enough that people remember it.

---

### Situation 6: You're Receiving the Feedback

**Context:** Someone reviewed *your* PR and you disagree.

**Bad way:**
```
No, it's fine like this.
```
❌ Closes the conversation without resolving it.

**Good way:**
```
Good catch on the null check, fixed.

On the second one — I kept the duplication on purpose because
the two flows are diverging next sprint. Happy to unify them
if you'd rather, though.
```
✅ Take the good point, explain the other with a reason, leave the door open.

---

### Situation 7: The PR Is Too Big to Review

**Context:** 2,000 lines. You cannot review this honestly.

**Bad way:**
```
[Skim it and approve]
```
❌ A rubber stamp is worse than no review.

**Good way:**
```
Heads up — this is a bit large for me to review properly. I've
gone through the API changes carefully; the migration I've only
skimmed. Could someone with more context take that part?
```
✅ Honest about your coverage instead of pretending.

## Phrases

### Marking severity

```
Blocking: ...
Non-blocking, but worth considering: ...
Nit: ...
Just a thought, ignore if you disagree: ...
Question, not a criticism: ...
Not for this PR, but eventually...
```

### Pointing out a problem

```
I think this breaks when...
This might throw if...
Small issue: ...
I could be wrong, but doesn't this return undefined when...?
Heads up — this conflicts with the change in X.
```

### Asking about edge cases

```
What happens if...?
Is it possible for X to be null here?
Have we thought about the case where...?
Does this handle an empty list?
What's the behaviour on timeout / retry / concurrent calls?
```

### Asking why (without sounding like an accusation)

```
Curious about the reasoning here.
What's the thinking behind this approach?
Is there a reason not to use X? Genuine question.
I might be missing context — why the second loop?
Help me understand this part.
```

### Suggesting

```
Would it be simpler to...?
Another option: ...
What do you think about extracting this into...?
Could we reuse the helper in utils for this?
```

### Approving

```
LGTM.
Looks good — nice work on X specifically.
Nice, this is much cleaner than the old version.
Approving. One nit inline, feel free to ignore.
```

### Responding to review on your PR

```
Good catch, fixed.
Fair point — updated.
Done in the last commit.
I kept it this way on purpose because... but happy to change it.
Not sure I agree — my thinking was X. What do you think?
Let's discuss this one on a call, easier than typing.
```

## Key Rules

1. **Mark severity first**
   - "Blocking" / "nit" / "question" tells them how to read everything after it.
2. **Comment on the code, not the person**
   - ❌ "You forgot the null check."
   - ✅ "This throws if user is null."
3. **"What happens if..." beats "did you test this?"**
   - Same question, no accusation.
4. **Ask when you don't understand**
   - You can't approve what you can't follow, and "this is hard to follow" is valid feedback.
5. **Say what's good, specifically**
   - It costs one sentence and changes how the rest is received.
6. **Move to a call after two rounds**
   - If it's still going back and forth, text is the problem. "Easier on a call?"
7. **Don't soften until the issue disappears**
   - A real bug stated politely is still a real bug. A real bug stated so gently it gets merged helped nobody.
8. **Reviewing is where written English is easiest**
   - You have unlimited time and can rewrite the sentence. Use that.

---

## What NOT to Do

| ❌ Don't write | ✅ Write instead |
|---|---|
| "This is wrong." | "Blocking: this throws when X is empty." |
| "Why did you do it this way?" | "Curious about the reasoning here — I might be missing context." |
| "Did you test this?" | "What happens if the request times out?" |
| "I'd use a map." | "Nit: a map might read cleaner. Fine as is." |
| "Maybe it's okay, I'm not sure..." | "Non-blocking, but I think this breaks on empty input." |
| "LGTM" (on a 2,000-line PR) | "I reviewed the API part carefully, skimmed the migration." |
| "No, it's fine." | "I did it this way because X — happy to change it though." |
| *Approving something you don't understand* | "Question: I'm not following the second loop." |

## The Bigger Picture

Code review turned out to be the easiest place to practise everything from the earlier blocks. Disagreeing (**Lesson 5**), proposing (**Lesson 6**), asking for clarification (**Lesson 2**) — all of it shows up in a PR, except here I get to edit the sentence before anyone sees it.

So I've started treating review comments as writing practice. If a comment sounds harsh, I rewrite it. If it sounds so soft the point is gone, I rewrite it the other way. Over a few weeks the good version starts arriving first.

That's the whole goal of this plan, really: not to learn new English, but to make the right version the one that comes out automatically. Written review is where I get to cheat and take my time.
