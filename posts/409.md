# Part 1 — The basics

## What Node actually is

Node is V8 (Chrome's JavaScript engine) plus **libuv**, a C library that provides the event loop and asynchronous I/O. That's the whole architecture, and understanding the split explains most of Node's behavior.

The model is **single-threaded, event-driven, non-blocking I/O**. One thread runs your JavaScript. When you ask for something slow — a file read, a database query, an HTTP request — Node hands it off and continues executing. When the result is ready, your callback gets queued.

The consequence people miss: **Node is excellent at I/O-bound work and bad at CPU-bound work.** Waiting on ten thousand database queries costs almost nothing, because waiting is free. Hashing ten thousand passwords blocks the one thread that serves every request.

As of 2026 the versions that matter are **Node 24 (Active LTS)** and **Node 22 (Maintenance LTS)**. Node 20 and 18 are both end-of-life — any advice written against Node 18 semantics is stale, and several classic interview answers are now simply wrong.

## Modules: CommonJS vs ESM

Node has two module systems, and the friction between them is a real interview topic.

```js
// CommonJS — the original
const express = require('express');
module.exports = { app };

// ESM — the standard
import express from 'express';
export { app };
```

The differences that matter:

- **`require` is synchronous and dynamic.** You can call it conditionally, inside a function, at any time.
- **`import` is static and hoisted.** The engine resolves the entire module graph *before* executing any module body. This is what enables tree-shaking — and what breaks Jest's mocking, as we'll see later.
- ESM has **top-level `await`**. CommonJS doesn't.
- In ESM there is no `__dirname` or `__filename`. Use `import.meta.dirname` (Node 20.11+) or derive it from `import.meta.url`.

You opt into ESM with `"type": "module"` in `package.json`, or the `.mjs` extension. CommonJS can't `require()` an ESM module synchronously (Node 22+ relaxes this for modules with no top-level await), but ESM can always `import` CommonJS.

New projects should use ESM. It's the standard, and the tooling has caught up.

## package.json and npm

- **`dependencies`** ship to production. **`devDependencies`** don't.
- **`package-lock.json`** pins the exact resolved tree. **Commit it.**
- **`npm ci`** installs exactly what the lockfile says and fails if `package.json` disagrees. **`npm install` can silently update things.** Use `ci` in CI — always.
- Semver ranges: `^1.2.3` allows minor and patch updates, `~1.2.3` allows patch only, `1.2.3` is exact.

That `^` is convenient and it's also how a compromised package reaches production in minutes. More on that in the security section.

## Callbacks, promises, async/await

The three generations of async in Node, in order.

**Callbacks** with the error-first convention:

```js
fs.readFile('file.txt', (err, data) => {
    if (err) return handleError(err);
    // ...
});
```

The error is always the first argument. Nesting these produces "callback hell", which is why promises exist.

**Promises** — an object representing a future value, with `pending`, `fulfilled` and `rejected` states.

**async/await** — syntax over promises. An `async` function always returns a promise; `await` pauses it until the promise settles.

```js
async function getUser(id) {
    try {
        const user = await repo.findById(id);
        return user;
    } catch (err) {
        logger.error({ err }, 'failed to load user');
        throw err;
    }
}
```

The concurrency operators worth knowing cold:

| Method | Behavior |
| --- | --- |
| `Promise.all` | Waits for all; **rejects immediately** if any rejects |
| `Promise.allSettled` | Waits for all; never rejects, returns status per promise |
| `Promise.race` | Settles with the first to settle, success or failure |
| `Promise.any` | Resolves with the first *success*; rejects only if all fail |

The classic mistake:

```js
// sequential — 3 seconds
const a = await fetchA();
const b = await fetchB();

// concurrent — 1 second
const [a, b] = await Promise.all([fetchA(), fetchB()]);
```

If two async operations don't depend on each other, awaiting them in sequence wastes the entire point of the runtime.

## Error handling

The rules, in order of how often they're violated:

**An unhandled promise rejection crashes the process** in modern Node (it was a warning before Node 15). This is deliberate — an unknown broken state is worse than a restart.

**`try/catch` does not catch errors in a callback.** The callback runs later, on a different stack:

```js
try {
    setTimeout(() => { throw new Error('boom'); }, 100);
} catch (e) {
    // never reached — the stack is long gone
}
```

**Always throw `Error` objects**, never strings. Only an `Error` carries a stack trace.

Two process-level nets, for logging before dying — not for continuing:

```js
process.on('uncaughtException', (err) => { logger.fatal({ err }); process.exit(1); });
process.on('unhandledRejection', (err) => { logger.fatal({ err }); process.exit(1); });
```

## EventEmitter

The pub/sub primitive that most of Node's own API is built on — streams, HTTP servers, and processes are all emitters.

```js
import { EventEmitter } from 'node:events';

class OrderService extends EventEmitter {
    complete(order) {
        this.emit('order:completed', order);
    }
}

service.on('order:completed', (order) => sendReceipt(order));
```

Two things interviewers check:

- **Listeners are synchronous.** `emit()` calls every listener in order and doesn't return until they've all run. An `async` listener isn't awaited — its rejection becomes an unhandled rejection.
- **An `'error'` event with no listener throws and crashes the process.** Always attach an error listener to an emitter.

The default max of 10 listeners per event isn't a limit, it's a leak warning — it usually means you're subscribing inside a request handler and never unsubscribing.

## Streams and buffers

A **Buffer** is a fixed-length chunk of binary data outside V8's heap. A **Stream** processes data in pieces instead of loading it all at once.

Four types: **Readable**, **Writable**, **Duplex** (both), and **Transform** (a duplex that modifies data passing through).

The reason streams matter, in one comparison:

```js
// loads the entire file into memory — a 2GB file is a 2GB allocation
const data = await fs.readFile('huge.csv');

// constant memory regardless of file size
await pipeline(
    createReadStream('huge.csv'),
    parseCsv(),
    createWriteStream('out.json')
);
```

**Use `pipeline()` from `node:stream/promises`, not `.pipe()`.** `.pipe()` does not forward errors or clean up the remaining streams when one fails, which leaks file descriptors. `pipeline` handles both.

## Core modules

`node:fs` (files — use `fs/promises`), `node:path` (path manipulation — always use it instead of string concatenation, for cross-platform correctness *and* to avoid traversal bugs), `node:http`, `node:crypto`, `node:os`, `node:url`, `node:stream`, `node:worker_threads`.

The `node:` prefix is worth adopting: it's explicit, and it can't be shadowed by a package of the same name from npm.

## Express fundamentals

```js
import express from 'express';

const app = express();
app.use(express.json());              // parse JSON bodies

app.get('/users/:id', async (req, res, next) => {
    try {
        const user = await service.find(req.params.id);
        if (!user) return res.status(404).json({ error: 'Not found' });
        res.json(user);
    } catch (err) {
        next(err);
    }
});
```

**Middleware** is a function `(req, res, next)` running in registration order. It either sends a response or calls `next()`.

**Error middleware takes exactly four arguments** `(err, req, res, next)`. Express identifies it by arity — drop the unused `next` and Express silently registers it as ordinary middleware, and errors never reach it. Don't let a linter talk you out of that parameter.

```js
app.use((err, req, res, next) => {
    if (res.headersSent) return next(err);
    req.log.error({ err }, 'request failed');
    res.status(err.status ?? 500).json({
        error: err.expose ? err.message : 'Internal Server Error'
    });
});
```

Never return the raw error message to the client by default — stack traces, SQL fragments and file paths are exactly what an attacker wants.

**Express 4 vs 5, and this is a live interview question:** in Express 4, an error thrown inside an `async` handler is **not** caught. The promise rejects, nothing forwards it, and the request hangs until it times out. That's why `express-async-errors` existed. **Express 5 fixed this natively** — rejected promises from async handlers go to the error handler automatically. A great deal of published advice still says the wrapper is mandatory; on Express 5 it isn't.

## Project structure and config

```
src/
├── routes/          # HTTP layer, validation
├── controllers/     # request → service → response
├── services/        # business logic
├── repositories/    # data access
├── middleware/
└── config/
```

Same layering as the Java and Go posts, same rule: dependencies point inward.

For configuration, Node 20+ has native `--env-file`, so `dotenv` is no longer needed. **Validate your environment at startup** — a config schema that fails fast on boot beats discovering a missing variable at 3am when the first request hits the code path that needed it.

# Part 2 — Advanced

## The event loop, properly

This is the defining Node interview question, and most explanations of it are wrong or vague.

The loop runs in **phases**, in this order, forever:

1. **timers** — callbacks from `setTimeout` and `setInterval` whose time has elapsed.
2. **pending callbacks** — some deferred system-level callbacks (certain TCP errors).
3. **idle, prepare** — internal.
4. **poll** — retrieve new I/O events, and execute their callbacks. **This is where Node spends most of its time.**
5. **check** — `setImmediate` callbacks.
6. **close callbacks** — `'close'` events, e.g. a destroyed socket.

Between **every** phase — and between each individual callback — Node drains two microtask queues:

- **`process.nextTick` queue** — drained first, completely.
- **Promise microtask queue** — drained next, completely.

That ordering is the thing to actually know:

```js
console.log('1');
setTimeout(() => console.log('2'), 0);
setImmediate(() => console.log('3'));
Promise.resolve().then(() => console.log('4'));
process.nextTick(() => console.log('5'));
console.log('6');

// 1, 6, 5, 4, 2, 3
```

Synchronous code first (`1`, `6`). Then `nextTick` (`5`), then promises (`4`). Then the loop starts its phases: timers (`2`), then check (`3`).

**`setTimeout(fn, 0)` vs `setImmediate(fn)` at the top level is genuinely non-deterministic** — it depends on how long the process took to start, which decides whether the first timers phase finds the timer expired. Inside an I/O callback, though, the order is guaranteed: `setImmediate` always wins, because the check phase comes right after poll, while timers requires wrapping all the way around.

**`process.nextTick` starves the loop if you recurse on it.** Because the nextTick queue is drained *completely* before moving on, an infinitely self-scheduling `nextTick` never lets the loop advance. `setImmediate` recursion is safe — it yields a phase each time. In application code, you almost never want `nextTick`.

## libuv and the thread pool

"Node is single-threaded" is true of your JavaScript and false of the runtime.

libuv maintains a **thread pool, 4 threads by default** (`UV_THREADPOOL_SIZE`, max 1024). Two categories of work get handled differently:

- **Network I/O** does *not* use the thread pool. It uses the OS's native async mechanisms — epoll on Linux, kqueue on macOS, IOCP on Windows. This is why Node handles tens of thousands of concurrent sockets on one thread.
- **Filesystem operations, DNS lookups (`dns.lookup`), and some `crypto` functions** *do* use the thread pool, because there's no portable async filesystem API.

The practical consequence: `crypto.pbkdf2` and `bcrypt.hash` occupy a pool thread. Fire five concurrent password hashes with the default pool of four and the fifth one queues — even though your event loop is idle. This is a favorite interview trap and a real production issue on login endpoints.

## Blocking the event loop

One synchronous slow operation stops *every* request in the process. Not just the current one — the entire server.

```js
// blocks everything for as long as it takes
const data = fs.readFileSync('huge.json');
const hash = crypto.pbkdf2Sync(pw, salt, 600000, 32, 'sha256');
JSON.parse(hugeString);                    // synchronous, no async version
array.sort(expensiveComparator);
new RegExp(userInput).test(long);          // ReDoS — catastrophic backtracking
```

`JSON.parse` on a large payload is the one people forget: there's no async version, and it's a common reason for latency spikes. Cap your request body size.

The fixes, in order of preference: use the async API; move the work to `worker_threads`; offload it to a separate service or a job queue; or break it into chunks with `setImmediate` between them.

## Measuring the event loop

"How would you know the event loop is blocked?" is the practical follow-up, and the answer is not a `setTimeout` drift loop — that competes for the very loop it's trying to measure.

```js
import { monitorEventLoopDelay } from 'node:perf_hooks';

const h = monitorEventLoopDelay({ resolution: 20 });
h.enable();

setInterval(() => {
    logger.info({ p99: h.percentile(99) / 1e6 }, 'event loop lag ms');
    h.reset();
}, 10_000).unref();
```

Note the `/ 1e6` — **the histogram is in nanoseconds**, and reporting `h.mean` straight into a dashboard labelled "ms" is a classic mistake.

Healthy numbers: mean lag under a millisecond, p99 under 10ms. Sustained p99 above 50–100ms means something is blocking, and remember what that means — a 200ms spike added 200ms to *every* request in flight.

The companion metric is **event loop utilization**, which measures saturation rather than delay:

```js
const elu = performance.eventLoopUtilization(previous);   // 0..1
```

Above roughly 0.7 sustained, latency is about to go non-linear. It's a good autoscaling signal, and a good load-shedding signal — fail the readiness probe when saturated so the load balancer routes elsewhere.

A useful connection to make in an interview: **a memory leak shows up as a latency problem first.** More live objects means more frequent major GCs, which means more pause time, which means rising p99 long before you hit an out-of-memory crash.

## worker_threads vs cluster vs child_process

Three ways to get more than one thread, for three different problems.

| | Use it for | Memory |
| --- | --- | --- |
| **`worker_threads`** | CPU-bound work inside one process | Shared process, can share memory via `SharedArrayBuffer` |
| **`cluster`** | Scaling an HTTP server across cores | Separate processes, no shared memory |
| **`child_process`** | Running a separate program or script | Fully separate |

**`cluster`** forks one process per CPU core, all sharing a listening port, with the OS or the primary distributing connections. It does not make a single request faster — it lets you serve N requests in parallel. Note that in a containerized deployment you often skip `cluster` entirely and run one process per container, letting the orchestrator handle scaling.

**`worker_threads`** is for the CPU-bound case: image processing, big data transforms, cryptography. Communication is by message passing, with `SharedArrayBuffer` if you need genuinely shared memory.

Important for cluster: **each worker is a separate process with its own memory**. In-memory rate limit counters, caches and sessions are per-worker and will behave unpredictably. That state has to live in Redis.

## Backpressure

The concept that separates people who've used streams from people who've read about them.

If a readable source produces data faster than the writable destination can consume it, the difference accumulates in memory. Backpressure is the mechanism that stops this: `writable.write()` returns `false` when its internal buffer is over the high water mark, and you're supposed to stop writing until the `'drain'` event fires.

```js
// ignores backpressure — memory grows without limit
source.on('data', chunk => destination.write(chunk));

// respects it
if (!destination.write(chunk)) {
    source.pause();
    destination.once('drain', () => source.resume());
}
```

`pipe()` and `pipeline()` handle this for you. Which is the real argument for using them: not brevity, correctness.

## Memory and the V8 garbage collector

V8's heap is **generational**, on the same bet the JVM makes — most objects die young.

- **New space (young generation)** — small, collected very frequently with a fast Scavenger. Cheap.
- **Old space** — objects that survived a couple of scavenges get promoted here. Collected by mark-sweep-compact, which is more expensive.

**Worth un-learning: there is no flat "Node caps the heap at 1.5GB" rule.** That was true years ago and is repeated everywhere. Modern V8 sizes the default old-space limit from available memory — roughly 2GB by default, up to about 4GB on machines with 16GB+ of RAM, and it reads the **cgroup limit** in a container rather than the host's RAM.

Two things still matter in production:

- **Set `--max-old-space-size` explicitly in containers**, at roughly 75–80% of the container limit. Heap is not RSS: Buffers, native addons and each worker thread's own heap live outside it. Sizing the heap equal to the container limit guarantees an OOM kill.
- **Node respects Kubernetes `limits`, not `requests`.** With only a request set, Node may size a heap far larger than the pod is allowed, and the kernel OOM-killer takes it (exit 137) before V8 ever tries hard to collect. The tell is diagnostic: `OOMKilled` with no stack trace means the kernel; `FATAL ERROR: Reached heap limit` means V8.

And raising the limit is not a fix for a leak — it moves the crash later and makes every major GC pause longer.

Worth correcting one more time, since it comes up: **V8's GC is not "stop-the-world for hundreds of milliseconds"**. That's 2015-era knowledge. Modern V8 marks concurrently on background threads and collects in parallel; minor GC is typically around a millisecond, and major GC's actual stop-the-world pause is a fraction of the total GC work. What it isn't is *free* — those helper threads still contend for CPU, which bites on a container limited to one core.

**Common leak sources**, roughly by frequency:

- **Listeners added per request and never removed.** The "MaxListenersExceededWarning" is usually this.
- **An unbounded cache or `Map`** used as a cache with no eviction. Use `WeakMap` when keys are objects whose lifetime you don't control, or a real LRU with a size cap.
- **Closures capturing large objects** — a small callback that keeps a huge request body alive.
- **Timers never cleared**, especially `setInterval` in a module that gets re-imported.
- **Globals and module-level arrays** that only ever grow.

To debug: take two heap snapshots with `--inspect` in Chrome DevTools, at times T1 and T2, and use the comparison view. What grew between them is your leak. `process.memoryUsage()` gives you the coarse numbers for alerting.

## Security

The section with the most outdated advice on the internet. I'll flag the specific things worth un-learning.

### Injection

Parameterized queries work because the SQL text is parsed into a query plan **before** the values arrive. Values are bound as typed literals and never lexed as SQL. That's categorically stronger than escaping.

```js
await client.query('SELECT * FROM users WHERE email = $1', [email]);
```

Three gotchas:

- **Identifiers cannot be parameterized.** Table names, column names and `ORDER BY` direction are not bindable. `ORDER BY $1` does not work. Use an allowlist map instead.
- **ORMs are not automatic immunity.** `knex.raw`, `$queryRawUnsafe` in Prisma, and `.where("id = " + id)` in TypeORM all reintroduce it. Prisma's tagged-template `$queryRaw` is safe; `$queryRawUnsafe` is not — the naming *is* the warning.
- In `mysql2`, `.execute()` is a real server-side prepared statement; `.query()` with `?` is client-side interpolation.

**NoSQL operator injection** is the Node-specific version, and it's asked constantly. `express.json()` gives you arbitrary JSON *objects*, and MongoDB filters are also objects where `$`-prefixed keys are operators:

```js
// VULNERABLE
await User.findOne({ email: req.body.email, password: req.body.password });
```

Send `{"email": {"$ne": null}, "password": {"$ne": null}}` and you've logged in as the first user in the collection. With `$regex` you can extract a password character by character.

The fix is **type validation at the boundary**, not sanitization:

```js
const Login = z.object({ email: z.string().email(), password: z.string().min(8) });
const { email, password } = Login.parse(req.body);   // an object is rejected outright
```

Two notes: Mongoose casting helps but isn't the control — untyped and `Mixed` paths still let operators through. And **`express-mongo-sanitize` is a trap answer** — it's effectively unmaintained, it breaks on Express 5 (where `req.query` became a getter), and blacklisting key prefixes is the wrong layer entirely.

### Prototype pollution

The Node-specific vulnerability class. Every object literal delegates to `Object.prototype`, so if a recursive merge writes an attacker-controlled key path without filtering `__proto__`, the attacker writes onto `Object.prototype` — visible on **every object in the process**.

```js
merge({}, JSON.parse('{"__proto__":{"isAdmin":true}}'));
({}).isAdmin;      // true — globally
```

It escalates past logic bypass to RCE when the polluted property is one Node itself reads later as an absent optional — `shell` or `env` in a `child_process.spawn` options object, or a template engine's compile options.

Defenses: `Object.create(null)` or `Map` for anything key-value from user input, `Object.hasOwn()` instead of `in`, schema validation with `additionalProperties: false`, and `node --disable-proto=throw`.

Worth correcting: **`JSON.parse` itself is safe.** It creates `__proto__` as a genuine own property and doesn't invoke the setter. The pollution happens in the merge or clone step afterward. `Object.assign` is shallow and therefore safe; deep merges are the risk.

### JWT

**Algorithm confusion** is the attack to understand. The server holds an RSA *public* key. The attacker re-signs the token with `alg: HS256`, using the PEM text of that public key as the HMAC secret. A verifier that dispatches on the token's own `alg` header takes the HMAC branch, HMACs with the same key material, and the signature validates. Full forgery from public information.

The fix is one line, and it's not optional:

```js
jwt.verify(token, publicKey, {
    algorithms: ['ES256'],        // explicit allowlist — without this you're vulnerable
    issuer: 'https://auth.example.com',
    audience: 'https://api.example.com',
});
```

Never let the token tell you how to verify the token. Related: `alg: none`, and `kid` header injection when the key ID is used in a path or a SQL lookup.

**Access vs refresh**: access tokens short (5–15 minutes), refresh tokens long-lived and **opaque random strings backed by a database row** — not JWTs. Making the refresh token opaque is deliberate, because it must be revocable, and statelessness is worthless for a credential you need to kill.

**Storage**: `localStorage` means XSS exfiltrates a portable bearer token that keeps working from the attacker's machine. An `httpOnly` cookie can't be read by JS but is auto-attached, so it needs `SameSite` and CSRF protection. Neither survives XSS — httpOnly just raises the cost. The strong pattern for SPAs is **access token in memory only, refresh token in an httpOnly cookie**, so nothing persistent is reachable by script.

**The revocation problem**, and the answer that signals seniority: a signed JWT is valid until `exp` by construction. "Log out everywhere" and "ban this user" don't work natively. You solve it with short TTLs plus **refresh rotation with reuse detection** — if an already-used refresh token is presented, two parties hold it, so you revoke the whole session family.

And the part worth volunteering: **if you have one backend and a browser client, server-side sessions with an opaque cookie are simpler and more secure than JWTs.** Instant revocation, no key management, no algorithm confusion. "We use JWT because it's stateless" is a red flag if the system then checks a Redis denylist on every request — that's a session with extra steps and a worse cache story. JWTs earn their place when verification must happen without contacting the issuer: independent services, third parties, OIDC federation.

### Rate limiting

**Token bucket** is usually the right algorithm for an API — it permits bursts up to the bucket size while bounding the sustained rate, and real clients are bursty. Fixed windows have a boundary flaw: 100/min allows 200 requests across one second at the window edge.

Two failure modes that make rate limiting silently useless:

**In-memory stores don't work with more than one process.** `express-rate-limit`'s default `MemoryStore` is per-process, so N instances behind a load balancer means N × your limit, and it resets on every deploy. Cluster mode counts too. Use `rate-limit-redis`.

**`trust proxy` is a trap in both directions.** Behind a load balancer, `req.ip` is the proxy's IP, so every user shares one bucket and one abuser rate-limits your whole userbase. But the naive fix is worse:

```js
app.set('trust proxy', true);    // ✗ now bypassable
app.set('trust proxy', 1);       // ✓ exactly one proxy hop
```

`X-Forwarded-For` is an append-only, client-controlled header. With `true`, Express trusts the whole chain and takes the leftmost entry — which the *client* wrote. An attacker sends a random value per request and gets unlimited fresh buckets. Set the number of proxies you actually control. `express-rate-limit` v7+ throws a validation error to catch exactly this, and disabling that validation to make the error go away is the wrong move.

Also: bucket IPv6 by /64 prefix, not full address. Rate-limit authenticated traffic by user ID rather than IP, since NAT lumps thousands of users together. And rate limiting is not DDoS protection — volumetric attacks have to die at the edge, before Node's event loop.

### Password hashing

Not SHA-256, and not salted SHA-256. They're designed to be *fast*, which is the entire problem — GPUs do billions per second. A salt defeats rainbow tables; it does nothing about throughput. You need a deliberately slow, **memory-hard** function, because memory-hardness is what destroys the GPU advantage: you can fit thousands of hash cores on a die, but you can't give each one 64 MiB of fast RAM.

Current OWASP parameters:

| Algorithm | Minimum |
| --- | --- |
| **Argon2id** (first choice) | m=19 MiB, t=2, p=1 |
| **scrypt** | N=2^17, r=8, p=1 |
| **bcrypt** (legacy) | work factor ≥ 10 |
| **PBKDF2** (FIPS only) | 600,000 iterations with SHA-256 |

**bcrypt silently truncates at 72 bytes.** Any two passwords sharing a 72-byte prefix are the same password — which matters for passphrases and password-manager output. And it's 72 *bytes*, so multi-byte UTF-8 hits the ceiling much sooner.

Do **not** naively pre-hash with SHA-256 to work around it. A raw binary digest can contain a null byte, and some bcrypt implementations truncate there, collapsing your entropy. There's also "password shucking": if you store `bcrypt(md5(pw))` and the attacker already has that MD5 from another breach, they crack the fast hash and shuck the bcrypt layer.

Also: always run the hash even when the user doesn't exist, or response timing leaks account existence. Store a **pepper** in a secret manager separate from the database, so a pure SQL dump is useless. And cap input length — a 1 MB password with Argon2id is a CPU bomb.

For Node in 2026, `@node-rs/argon2` avoids the native-toolchain pain that makes the `bcrypt` package a recurring CI headache.

### CORS

CORS is a *relaxation* of the same-origin policy, not a security control. Every mistake means "I let someone else's website read my authenticated responses."

The one to know:

```js
app.use(cors({ origin: true, credentials: true }));   // ✗ this is the vulnerability
```

`origin: true` does not mean "same origin". It means **echo the request's `Origin` header back**. With `credentials: true`, any website can now make credentialed requests as your logged-in user *and read the response*. It appears constantly in tutorials and AI-generated code as the fix for CORS errors. It is the bug.

```js
const ALLOWED = new Set(['https://app.example.com']);
app.use(cors({
    origin: (origin, cb) =>
        !origin || ALLOWED.has(origin) ? cb(null, origin ?? false) : cb(new Error('CORS')),
    credentials: true,
}));
```

Bad allowlist checks to avoid: `.endsWith('example.com')` matches `notexample.com`; `.includes()` matches `example.com.evil.io`. Never allowlist `null` — sandboxed iframes send it. And set `Vary: Origin` or a shared cache will serve one origin's header to another.

Two things people conflate: **CORS is enforced by the browser only** — curl and any server ignore it entirely, so it's never authorization. And "simple requests" skip preflight, which means the request **reaches your handler and any side effect happens** even if the browser then blocks reading the response. A state-changing GET is exploitable regardless of your CORS config.

### helmet

Sets 13 security headers and removes `X-Powered-By`. Worth knowing what it actually does, because one default surprises people:

**helmet sets `X-XSS-Protection: 0`, deliberately.** Not `1; mode=block`. The legacy browser XSS auditor was buggy and introduced vulnerabilities of its own; every major browser removed it; explicitly sending `0` disables any residual filter. A large amount of published security-checklist content still tells you to set `1; mode=block`, and that advice is backwards.

Also: **CSP's `frame-ancestors` supersedes `X-Frame-Options`** in modern browsers. And on a pure JSON API, most of helmet is theater — CSP does little for a response no browser renders. The load-bearing headers for an API are `nosniff`, HSTS, and the cross-origin policies. Knowing *which* ones matter is the better answer.

### Supply chain

This became the headline Node risk in 2025, and the OWASP Top 10 was restructured around it — supply chain failures are now their own category rather than "vulnerable and outdated components".

Two incidents worth being able to name. In September 2025, a maintainer was phished and 18 packages including `chalk` and `debug` — roughly 2.6 billion weekly downloads combined — shipped malicious versions. A week later, **Shai-Hulud** was qualitatively worse: a self-replicating worm whose `postinstall` script harvested npm tokens, GitHub PATs and cloud credentials from developer machines, then used the stolen npm token to publish itself into that maintainer's own packages. 500+ packages were removed.

The mitigations that actually address this:

```
# .npmrc
ignore-scripts=true          # kills the postinstall vector — highest-value single control
```

```jsonc
// package.json, npm 11.5+
"minimumReleaseAge": "1440"  // refuse packages published in the last 24h;
                             // nearly all these attacks are caught and yanked within hours
```

Plus: commit the lockfile and use `npm ci`; pin exact versions, because `^` is what let those downloads move in minutes; provenance and trusted publishing via OIDC instead of long-lived tokens; and scoped read-only CI tokens.

**`npm audit` is a floor, not a control.** It only reports publicly disclosed advisories — it is completely blind to a *fresh* malicious version, which is the entire attack window. It has no reachability analysis, so it flags vulnerabilities in code paths you never call, and `npm audit fix --force` will install breaking majors. What catches these attacks is behavioral tooling that flags a *new* install script or *new* network access in a dependency diff.

And remember `process.env` is readable by **any** dependency in the process. That's the whole Shai-Hulud thesis, and the reason secrets belong in a secret manager with rotation, not just in an environment variable you feel good about.

## Testing

**Vitest is the default recommendation for new Node projects in 2026.** Native ESM and TypeScript with no transform configuration, fast, and a Jest-compatible API so migration is mostly `jest.` → `vi.`. Jest remains reasonable for large existing suites. `node:test` — stable since Node 20 — is genuinely attractive for libraries that want **zero dev dependencies**, which, given the section above, is a defensible security argument and not just minimalism.

**Why Jest struggles with ESM**, since it's a good interview question: in CommonJS, Babel *hoists* `jest.mock()` above the `require` calls, so the mock registers before the module loads. Native ESM resolves all static imports before any module body runs, so hoisting is impossible. You need `jest.unstable_mockModule()` plus dynamic `await import()` afterward, and `__mocks__` directories stop auto-loading. Vitest's `vi.mock` *is* hoisted, because Vite transforms the module graph, so ordinary static imports just work. Jest 30 improved this materially; it's still not at parity, and the API is still named `unstable_`.

**The app/server split** is the pattern for testing HTTP:

```js
// app.js — routes and middleware, NO listen
export const app = express();

// server.js — the only place that listens
app.listen(process.env.PORT ?? 3000);

// app.test.js
const res = await request(app).post('/users').send({ email: 'a@b.com' }).expect(201);
```

Supertest calls `.listen(0)` internally on an ephemeral port and closes it. If your test imports a file that calls `listen()` at module scope, you get a real long-lived server, port conflicts under parallel runs, and an open handle that stops the process exiting.

**Don't mock the database.** A mock encodes your belief about the database, so the test can only confirm your belief. It can't fail on what actually breaks: a migration that didn't run, a constraint violation, `NULL` ordering, cascade deletes, isolation levels, pool exhaustion. It also couples tests to implementation — refactor two queries into one JOIN, identical behavior, every test breaks. That's a suite that punishes refactoring.

**Testcontainers** is the current answer — a real Postgres in Docker, started programmatically:

```js
const container = await new PostgreSqlContainer('postgres:17-alpine').start();
process.env.DATABASE_URL = container.getConnectionUri();
```

Start one container per run, migrate once, then isolate per test with a **transaction rollback** (fastest, perfectly isolated, but breaks if the code under test commits) falling back to `TRUNCATE ... RESTART IDENTITY CASCADE`. Do still mock third-party HTTP — Stripe, S3, SendGrid — with `nock` or `msw`, since those are slow, flaky and cost money.

Two more gotchas worth having ready:

**Un-awaited async assertions pass while asserting nothing.**

```js
it('rejects', () => { expect(fn()).rejects.toThrow(); });               // ✗ always passes
it('rejects', async () => { await expect(fn()).rejects.toThrow(); });   // ✓
```

The un-awaited rejection then surfaces in a *later, unrelated* test, which is miserable to debug.

**`--forceExit` is not a fix for a hanging test run**, it's suppression. The hang means something still holds the event loop open — an unclosed pool, a Redis client, a `setInterval`. Use `--detectOpenHandles` to find it and close it in `afterAll`. Forcing exit hides a resource leak that will reappear in production as pool exhaustion.

## Databases and connection pooling

**Why pool**: a Postgres connection costs a TCP handshake, TLS, auth, and a forked backend process — tens of milliseconds and several MB of RSS each. A pool amortizes that and, more importantly, **bounds** concurrency against the database.

**Sizing is counterintuitive: more is not better.** The pool exists to *limit* concurrency, not maximize it. Past the point where the database can actually execute queries in parallel, extra connections only add context switching and lock contention — throughput goes down. The classic heuristic is `(cores × 2) + spindles`, which lands in the low tens, not the hundreds. Little's Law gets you there too: 500 rps × 10ms per query ≈ 5 connections.

What matters is the **sum across every instance**. Twenty pods with a pool of twenty is 400 connections against a database configured for 100, and you discover it during a rolling deploy when old and new pods overlap.

Two configuration gotchas worth knowing cold:

```js
const pool = new pg.Pool({
    max: 10,
    connectionTimeoutMillis: 5_000,   // default is 0 = wait forever
    statement_timeout: 10_000,
});
pool.on('error', (err) => logger.error({ err }, 'idle client error'));
```

`connectionTimeoutMillis` defaults to **wait forever**. When the pool is exhausted, requests queue silently and unboundedly — latency climbs, nothing appears in the logs, and the event loop looks idle. A fast timeout is far better than an invisible queue. And you must attach `pool.on('error')`: an idle connection killed by a failover emits an error on the pool, and unhandled, it crashes the process.

Monitor `pool.waitingCount`. Persistently above zero is your saturation signal, and it's the answer to "the database looks fine but the app is slow".

**The transaction gotcha**, which is the most-asked pooling question:

```js
// ✗ each pool.query() may take a DIFFERENT connection.
await pool.query('BEGIN');
await pool.query('INSERT ...');    // possibly a different connection — auto-committed
await pool.query('COMMIT');        // and a third one
```

Transaction state is scoped to a connection. This gives you no atomicity at all, *and* leaves a connection sitting idle-in-transaction holding locks. You must check out one client and use it throughout:

```js
const client = await pool.connect();
try {
    await client.query('BEGIN');
    await client.query('UPDATE accounts SET balance = balance - $1 WHERE id = $2', [amt, from]);
    await client.query('UPDATE accounts SET balance = balance + $1 WHERE id = $2', [amt, to]);
    await client.query('COMMIT');
} catch (err) {
    await client.query('ROLLBACK').catch(() => {});
    throw err;
} finally {
    client.release();      // always — a missed release permanently costs you a connection
}
```

ORMs hand you a scoped client (`prisma.$transaction(async (tx) => ...)`), and the equivalent bug is using the outer handle instead of `tx` inside the callback, which silently runs that statement outside the transaction.

**Keep transactions short.** Never `await` an HTTP call inside one — you're holding both a pool slot and database locks.

**N+1 queries** are the same problem described in the SQL post, and ORMs generate them by default through lazy relations. It looks fine with ten rows in development and melts with ten thousand in production. Detect it by counting queries per request; fix it with eager loading, a JOIN, or `DataLoader` for GraphQL. Two things interviewers check about DataLoader: the batch function **must return an array matching the input keys in order and length** (returning DB rows directly is the classic bug, since they come back unordered and omit misses), and you must **create a new instance per request** — a module-level singleton has an unbounded cache and serves one user's data to another. That last one is a security answer as much as a performance one.

**Caching** follows the SQL post's advice, with one Node-flavored addition: **cache stampede**. A hot key expires, five thousand concurrent requests all miss, and they all hit the database at once — worse the more popular the key is, so it hits at peak traffic. Mitigations, in ascending order: TTL jitter (so keys don't expire in synchronized waves), a distributed lock so one request repopulates while the rest serve stale, and in-process request coalescing:

```js
const inflight = new Map();
function once(key, fn) {
    if (!inflight.has(key)) inflight.set(key, fn().finally(() => inflight.delete(key)));
    return inflight.get(key);
}
```

That last one is cheap, needs no Redis round trip, and handles the single-instance case entirely.

## Graceful shutdown

Rarely studied, frequently asked, and the source of the 502s everyone sees during deploys.

The sequence on `SIGTERM`: fail the readiness probe → wait for that to propagate → stop accepting new connections → drain in-flight requests → close the database pool and queue consumers → flush logs and traces → exit, with a hard timeout that force-exits.

**The Kubernetes race** is the part most implementations get wrong. Kubernetes sends `SIGTERM` and removes the pod from Service endpoints *at the same time*, and endpoint removal propagates asynchronously to every node. For several seconds after `SIGTERM`, **traffic is still being routed to you.** Close the server immediately and you reject it. The fix is a `preStop` sleep plus flipping readiness to failing before you close anything.

**The `server.close()` gotcha, and a correction worth making**: `close()` stops accepting new connections but only fires its callback once existing connections end. With HTTP keep-alive — the default in Node 19+ — idle-but-open connections never end on their own, so the callback may never fire and you hang until `SIGKILL`. A great deal of published code shows `server.close(() => process.exit(0))` with no timeout, and that code hangs in any real environment.

The fix has been built in since Node 18.2, so **the old advice to manually track every socket in a `Set` is obsolete**:

- `server.closeIdleConnections()` — destroys sockets currently between requests, leaving in-flight ones to finish. This is the correct drain primitive.
- `server.closeAllConnections()` — destroys everything. Failsafe, on timeout only.

```js
async function shutdown(signal) {
    if (shuttingDown) return;              // idempotent — SIGTERM can arrive twice
    shuttingDown = true;                   // readiness now returns 503

    await sleep(10_000);                   // let the load balancer notice

    const closed = new Promise((res) => server.close(res));
    server.closeIdleConnections();

    const timer = setTimeout(() => server.closeAllConnections(), 20_000).unref();
    await closed;
    clearTimeout(timer);

    await worker.close();                  // stop taking jobs, finish current
    await pool.end();
    await redis.quit();
    await logger.flush();

    process.exit(0);
}
```

**Order matters**: close the HTTP server *before* the database pool, or in-flight requests fail with "cannot use a pool after calling end". And `terminationGracePeriodSeconds` must exceed your preStop sleep plus your drain timeout, or Kubernetes kills you mid-drain.

Two more that are almost always the real cause when "my container ignores SIGTERM": **`node` as PID 1** in Docker gets no default signal handling — use `--init` or `exec node` in your entrypoint, because a shell wrapper without `exec` swallows the signal entirely. And `npm start` does not forward signals reliably; invoke node directly.

## Observability

**Structured JSON logging**, one object per line, never string concatenation. **pino** over winston, and the reason is structural rather than micro-optimization: pino uses pre-compiled serializers and moves formatting and transport to a **worker thread**, so logging doesn't tax the event loop, while winston's transports and format combinators run in-process on the main thread per line. Under load that difference is measurable in request latency.

Watch the argument order — it's `logger.info({ obj }, 'message')`, object first. The other way round silently discards your structure.

**Correlation IDs with `AsyncLocalStorage`** is the senior topic here. The problem: a request ID needs to appear on every log line across dozens of async functions, and you can't use a module-level variable because Node interleaves concurrent requests — by the time an `await` resumes, another request has overwritten it. Threading a `ctx` argument through everything is invasive.

`AsyncLocalStorage` gives you the equivalent of thread-local storage, propagated automatically across `await`, promises, callbacks and timers:

```js
export const als = new AsyncLocalStorage();

app.use((req, res, next) => {
    als.run({ requestId: req.get('x-request-id') ?? randomUUID() }, next);
});

// bind it into the logger and every line gets it for free
const logger = pino({ mixin: () => ({ requestId: als.getStore()?.requestId }) });
```

**Another piece of outdated advice worth flagging**: "async_hooks is too slow for production" was fair around Node 12–16. After V8's fast-path work for async context propagation it's consistently under about 4% on real HTTP workloads, and effectively zero when no store is active.

The gotchas: use **one instance per process** (instance count drives cost far more than store size); context is lost across `EventEmitter` listeners registered outside the `run()` — `AsyncResource.bind()` is the fix, and it's the answer to "why is my request ID undefined here"; don't store `req`/`res` or anything large, since the store is retained for the whole async tree; and prefer `als.run()` over `enterWith()`, which has no scope boundary.

**Metrics discipline**: **RED** (Rate, Errors, Duration) for the request path, **USE** (Utilization, Saturation, Errors) for every resource behind it. The framing: *RED tells you users are unhappy, USE tells you why.* Always percentiles for duration, never averages. Node-specific must-haves: event loop delay percentiles, event loop utilization, heap used against the limit, and pool waiting count.

**Liveness vs readiness**, and the gotcha interviewers reliably probe:

| | Liveness | Readiness |
| --- | --- | --- |
| Asks | Is the process wedged? | Can I serve traffic now? |
| On failure | Pod is **restarted** | Pod is removed from the load balancer |
| Should check | Almost nothing | DB, Redis, shutdown flag |

**Never check the database in your liveness probe.** A brief database hiccup fails liveness on every pod simultaneously, Kubernetes restarts your entire fleet, and you've turned a blip into an outage with a thundering herd of cold starts. Dependency checks belong in readiness, which removes you from the load balancer instead of killing you.

## The short version

1. **The event loop has phases, and microtasks drain between every callback** — not just at phase boundaries. That changed in Node 11, and the old diagrams are still everywhere.
2. **Filesystem, DNS lookups and crypto use libuv's 4-thread pool; network I/O doesn't.** That's why concurrent password hashing queues while the loop sits idle.
3. **`origin: true` with credentials is an open door**, not a CORS fix.
4. **Explicit `algorithms` in `jwt.verify`**, or algorithm confusion forges tokens from your public key.
5. **`ignore-scripts=true` and a release-age delay** are the two cheapest supply-chain controls you can adopt today.

And a bonus list, since this post spent a lot of time on it — **advice to un-learn**: the 1.5GB heap cap (obsolete), `.pipe()` (use `pipeline()`), manual socket tracking on shutdown (built in since Node 18.2), `X-XSS-Protection: 1; mode=block` (helmet disables it deliberately), "async_hooks is too slow" (under 4% now), and "bigger connection pool means more throughput" (it's inverted).
