There are 23 Gang of Four patterns and you will use about eight of them regularly. This is a cheat sheet weighted by how often each one shows up in real backend work, not by the order they appear in the book.

**The basics** covers the GoF patterns worth knowing, grouped the standard way. **The advanced part** covers the application-level patterns that aren't in the book but are in every codebase, plus the architectures they add up to.

One thing first, because it's the actual test:

> **A pattern is an answer to a problem. Without the problem, it's over-engineering.**

Naming twelve patterns is worth less than explaining why you *didn't* use one. If you take a single sentence from this post, take that one.

# Part 1 — The Gang of Four patterns

## Creational — how objects get made

### Singleton

One single global instance — a config object, a connection pool, a logger.

**Be careful with this one.** It's the most-taught and least-recommended pattern. It's global mutable state, it makes testing hard (you can't swap it out, and state leaks between tests), and it hides dependencies — a class using a singleton doesn't declare that it needs it.

In practice, an injected bean is almost always better. Spring's default bean scope *is* singleton, but managed by the container, which means you can still replace it in a test. Same lifetime, none of the problems.

### Factory Method / Factory

Delegates creation to a method or class, so the caller doesn't know the concrete type. Useful when the type to create depends on a parameter.

```java
interface PaymentProcessor { void charge(Money amount); }

class PaymentProcessorFactory {
    static PaymentProcessor forCountry(String country) {
        return switch (country) {
            case "AR" -> new MercadoPagoProcessor();
            case "US" -> new StripeProcessor();
            default   -> throw new UnsupportedCountryException(country);
        };
    }
}
```

The caller asks for a processor and gets one. It never learns which.

### Builder

Builds a complex object step by step, especially with many optional parameters. It avoids telescoping constructors.

```java
User.builder()
    .name("Juan")
    .age(25)
    .build();
```

Instead of:

```java
new User("Juan", 25, null, null, true);
```

The advantages:

- Easier to read — you can see what each argument means.
- Optional values are actually optional.
- No enormous constructors.

In Java it's usually Lombok:

```java
@Builder
class User {}
```

Go doesn't have Builder as such. The idiom there is a `NewX()` constructor with functional options:

```go
func NewServer(opts ...Option) *Server
server := NewServer(WithPort(8080), WithTimeout(30*time.Second))
```

> **Builder = assemble an object piece by piece, then call `build()`.**

### Prototype

Creates new objects by copying an existing one instead of building from scratch. Useful when construction is expensive or when you want to clone a configured template. The rarest of the four in day-to-day backend work.

## Structural — how objects are composed

### Adapter

Translates one interface into the one the client expects. You wrap an incompatible API.

> "I translate an incompatible interface."

This is what you write when a third-party SDK doesn't match your domain interface. Your code depends on `PaymentProcessor`; the adapter wraps the vendor SDK and makes it satisfy that interface. When the vendor changes, you rewrite the adapter and nothing else.

### Decorator

Adds behavior by wrapping an object, without modifying its class — logging, caching, retries, metrics around a service.

> "I add things around it without touching the class."

```java
class CachedUserRepository implements UserRepository {
    private final UserRepository inner;
    private final Cache cache;

    public User findById(String id) {
        return cache.get(id, () -> inner.findById(id));
    }
}
```

Same interface, extra behavior, and the original class never learns caching exists. You can stack them: cached, then logged, then retried.

### Facade

A simple interface hiding a complex subsystem. One method that orchestrates six calls, so callers don't have to know the sequence.

### Proxy

A stand-in that controls access to the real object — lazy loading, access control, caching.

> "I put an intermediary that controls access."

The distinction people get asked about: a **middleware** intercepts a request to the application; a **proxy** intercepts any call to that object, wherever it comes from.

Hibernate's lazy loading is a proxy: the object you hold isn't the entity, it's a stand-in that fetches the real one on first access. That's also why you get `LazyInitializationException` — the proxy outlived the session it needed.

### Composite

Treats individual objects and groups of objects uniformly, typically for trees.

> "I treat one object and a group of objects the same way."

Composite keeps a common interface between simple objects and groups of objects, so you can call the same methods without knowing which is behind it. A file and a folder both have `size()`; the folder's implementation sums its children.

## Behavioral — how objects interact

### Strategy

Encapsulates interchangeable algorithms behind an interface, chosen at runtime. **The favorite for replacing type-based `if`/`switch` chains** — fee calculation strategies, pricing rules, export formats.

This is the pattern behind the Open/Closed principle in practice, and probably the one you'll use most.

### Observer

A subject notifies several observers when it changes — events, pub/sub.

> "How do I tell others that something changed?"

It's like Command but **in memory**: inside a single program, using RAM and direct calls. Which means if it fails, the notification is lost. There's no retry and no durability. That's the key limitation to state out loud, because it's exactly what distinguishes it from a real message queue.

### Command

Encapsulates an action as an object, which lets you queue it, undo it, or log it. You can back it with Kafka or a job queue — **the focus is on the action**.

> "There's a pending action: send an email."

Observer answers *something changed*. Command answers *something must be done*. Observer is in-memory and lossy; Command can be persisted, retried, and replayed. Picking the right one is a real architectural decision.

### Template Method

The base class defines the skeleton of an algorithm and subclasses fill in the steps.

> Question it answers: "what's the flow of this process?"

Making coffee:

1. Heat the water
2. **Prepare the ingredient** ← this varies
3. Serve

Steps 1 and 3 live in the base class; step 2 is abstract. The flow is fixed, the details vary.

Note this one uses inheritance, which is why Strategy is often preferred — Strategy achieves something similar through composition and stays more flexible.

### State

An object changes its behavior depending on its internal state, without giant `if` chains. An `Order` behaves differently when it's `Pending`, `Paid`, or `Shipped`, and each state knows which transitions are legal.

# Part 2 — Advanced

## Application patterns — not GoF, but always present

These aren't in the book, and you'll use them more than anything that is.

**Repository** — abstracts persistence behind an interface. The Service doesn't know whether it's MySQL, Postgres or a mock, so you can test with an in-memory repo and change engines without touching business logic.

**DTO (Data Transfer Object)** — the structure that goes in and out of the API, decoupled from the domain entity. It controls which fields are exposed (you don't leak `passwordHash`) and decouples your API contract from your database schema. Without it, renaming a column is a breaking change for every client.

**Service layer** — concentrates business logic and orchestration: calls several repositories, manages the transaction boundary.

**Dependency Injection** — dependencies are passed in from outside rather than created inside. The practical application of the Dependency Inversion Principle. Prefer constructor injection: it makes dependencies explicit and immutable, and makes testing easy.

**Mapper** — converts Entity ↔ DTO in one place instead of scattered across controllers.

## Layered architecture

```
Controller/Handler  →  Service  →  Repository  →  Model/Entity
   (HTTP, validates      (business    (data          (domain
    input, maps DTOs)     logic)       access)        data)
```

**The rule: dependencies point inward.** The Controller knows the Service; the Service knows the Repository; never the reverse. Each layer exposes an interface to the one above it, which is what lets you test and replace implementations.

If you find yourself importing a controller from a service, something has gone wrong in a way that will get worse.

## Larger architectures

**Monolith vs microservices.** A monolith is one deploy — simple at the start, couples everything over time. Microservices are small independent services that scale separately, but add real complexity: network calls, distributed consistency, observability, deployment coordination. **Don't start with microservices without needing them.** Most teams that regret their architecture regret this direction, not the other one.

**Hexagonal / Ports & Adapters.** The domain sits at the center, isolated from details (database, HTTP, frameworks) through ports (interfaces) and adapters (implementations). Makes testing and infrastructure changes cheap.

**Clean Architecture.** Concentric layers with dependencies pointing at the domain. The business logic doesn't depend on frameworks. Same instinct as hexagonal, drawn differently.

**Event-driven.** Components communicate through events and messages (Kafka, RabbitMQ), decoupling producers from consumers. You gain independence and lose the ability to follow the flow by reading the code.

**CQRS** — separate the read model from the write model when they have genuinely different needs. Powerful, and frequently applied to systems that didn't need it.

## Concurrency concepts

Patterns interviews drift into concurrency constantly. The vocabulary:

- **Process vs thread** — a process has its own memory; threads share the process's memory. Lighter, but they need synchronization.
- **Concurrency vs parallelism** — concurrency is handling several tasks interleaved (they make progress in turns); parallelism is literally executing them at the same time on multiple cores.
- **Race condition** — the result depends on execution order over shared state. Avoided with synchronization, atomics, or immutability.
- **Deadlock** — tasks waiting on resources another holds and never releases. Prevention: acquire locks in a consistent order, use timeouts.
- **Blocking vs non-blocking** — blocking means the thread stands still and waits; async means the thread continues and gets notified when the work finishes (callbacks, promises, futures).
- **CPU-bound vs I/O-bound** — CPU-bound work speeds up with more cores; I/O-bound work (network, disk) benefits from async because the time is spent waiting, not computing.
- **Idempotency** — an operation produces the same result if repeated. This is what makes retries safe in distributed systems, and it's asked about constantly.

## HTTP, REST and API design

- **HTTP is stateless** — each request is independent. State lives in tokens, cookies or the database, not in the server.
- **Methods**: GET (read, safe and idempotent), POST (create, not idempotent), PUT (replace, idempotent), PATCH (partial update), DELETE (idempotent).
- **Status codes**: 2xx success (200 OK, 201 Created, 204 No Content), 3xx redirect, 4xx client error (400 Bad Request, 401 unauthenticated, 403 forbidden, 404 not found, 409 conflict, 429 too many requests), 5xx server error (500, 502, 503).
- **REST** — resources identified by URL, operations through HTTP methods, representations usually in JSON, stateless. Good practice: plural nouns (`/users/1`), no verbs in the URL.
- **Authentication vs authorization** — authentication is who you are (login); authorization is what you can do (roles and permissions).
- **JWT** — a signed token carrying claims. The server validates it without storing a session. Don't put sensitive data in it, and keep expiry short.
- **Idempotency and retries** — design write endpoints so they can be retried without duplicating, using idempotency keys.
- **Rate limiting, pagination and versioning** (`/v1/`) are standard API design topics.
- **CORS** — a browser mechanism controlling which origins may call your API. Note it's enforced by the browser, not by your server, so it is not a security boundary.

## Caching

Store expensive results to reuse them — lower latency, less load.

**Where**: in-process memory, distributed (Redis, Memcached), CDN (static content), or HTTP-level (`Cache-Control` / `ETag` headers).

**Strategies**: cache-aside (read the cache, on a miss go to the database and populate it), write-through, and TTL (expires on its own).

The hard problem is **invalidation** — keeping the cache coherent with the source of truth. As the joke goes: there are two hard things in computer science, naming things, cache invalidation, and off-by-one errors.

## Data structures and Big-O

Big-O describes how time or memory scales with input size, in the worst case.

| Complexity | Example |
| --- | --- |
| O(1) | Hash map lookup |
| O(log n) | Binary search, balanced tree |
| O(n) | Traversing a list |
| O(n log n) | A good sort |
| O(n²) | Nested double loop |
| O(2ⁿ) | Exponential brute force |

- **Array / list** — index access O(1), search O(n).
- **Hash map** — get/put O(1) average.
- **Balanced tree** — O(log n).
- **Set** — membership O(1) average.
- **Stack** (LIFO) and **Queue** (FIFO) — choose based on the operation you do most.

**Pure function** — no side effects, always returns the same output for the same input. Trivially testable.

**Recursion vs iteration** — recursion is elegant for trees and divide-and-conquer but consumes stack space (overflow risk) and needs a base case.

## Testing

- **The test pyramid** — many unit tests (fast, isolated), fewer integration tests, very few end-to-end tests (slow, fragile).
- **Unit test** — one unit in isolation, dependencies mocked. **Integration** — several pieces together (service + real database). **E2E** — the whole flow as a user.
- **Mock vs stub vs fake** — a stub returns fixed responses; a mock verifies interactions (that something was called); a fake is a lightweight working implementation (an in-memory repository).
- **TDD** — red (a failing test) → green (minimum code to pass) → refactor.
- **A good test** is fast, deterministic (not flaky), independent, and has one reason to fail. **AAA**: Arrange, Act, Assert.
- **Coverage** is a signal, not a target. 100% coverage does not mean well tested — it means every line ran, not that any behavior was verified.

## Git fundamentals

- **merge vs rebase** — merge joins histories with a merge commit and preserves what really happened; rebase rewrites your commits onto a new base for a linear history. **Don't rebase anything already pushed and shared.**
- **branch** — a movable pointer to a commit. Feature branches plus pull requests is the standard flow.
- **reset vs revert** — `git reset` moves the branch (`--soft`/`--mixed`/`--hard`); `git revert` creates a new commit that undoes a previous one, which is the safe option on shared branches.
- **staging area** — the intermediate zone between working directory and commit (`git add`).
- **stash** — save uncommitted changes to switch context.

## Quick answers

Short, correct responses to the questions that come up most:

- **SQL vs NoSQL?** SQL is relational, fixed schema, ACID, joins. NoSQL is flexible schema, horizontal scaling, high volume or unstructured data.
- **Abstract class vs interface?** An interface is a pure contract with no state and you can implement several; an abstract class can hold state and concrete methods, and you only extend one.
- **Overload vs override?** Overload: same name, different parameters, resolved at compile time. Override: a subclass redefines a base method, resolved at runtime.
- **Runtime vs compile-time polymorphism?** Runtime: override plus dynamic dispatch. Compile-time: overloading and generics.
- **Why composition over inheritance?** Inheritance couples tightly and creates fragile hierarchies; composition is flexible and testable.
- **What's a memory leak?** Memory that stays allocated and unreachable-but-referenced, so it's never freed — dangling references, listeners never removed.
- **Stateless vs stateful?** Stateless keeps no state between requests and scales easily; stateful does (sessions) and is harder to scale.
- **Optimistic vs pessimistic locking?** Optimistic: assume no conflict, verify at the end with a version. Pessimistic: lock the resource while working.
- **Synchronous vs asynchronous?** Sync: I wait for the response. Async: I continue and get notified when it's done.
- **Compiled vs interpreted?** Compiled (Go, C) translates to machine code before running — faster. Interpreted (Python) executes line by line. Java is a hybrid: bytecode plus a JIT-compiling JVM.

## The short version

1. **Strategy is the pattern you'll use most.** It's how Open/Closed looks in real code.
2. **Singleton is the pattern you'll use least.** Inject a bean instead.
3. **Observer is in-memory and lossy; Command can be persisted and retried.** Know the difference.
4. **Repository, DTO, Service, DI and Mapper** aren't in the book and are in every codebase.
5. **Dependencies point inward.** That single rule is most of what layered, hexagonal and clean architecture are saying.

And again, the one that matters: **without the problem, a pattern is over-engineering.** Show judgment, not a catalog.
