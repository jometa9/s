# 1. Clarify the Requirements (Always Start Here)

Never start drawing boxes immediately.

Spend the first few minutes asking questions.

### Functional Requirements

* Who are the users?
* What problem are we solving?
* What are the core features?
* Is authentication required?
* Is it web, mobile, desktop, or API?
* Is real-time communication required?
* What is out of scope?

### Non-Functional Requirements

* Expected latency?
* Availability target?
* Consistency requirements?
* Durability?
* Security?
* Scalability?
* Cost constraints?

### Scale Estimation

* Daily Active Users (DAU)
* Monthly Active Users (MAU)
* Requests per second (RPS)
* Read vs Write ratio
* Average object size
* Storage growth
* Peak traffic

---

# 2. Estimate the Scale

Quick calculations help justify design decisions.

Estimate:

* Users
* Requests/second
* Storage
* Network bandwidth
* Peak load
* Cache size

Example:

* 10M users
* 1M DAU
* 20 requests/day
* ≈230 requests/sec average
* 2,000+ requests/sec peak

# 5. Database Design

Choose the right database.

### SQL

Best for:

* Financial systems
* Transactions
* Strong consistency
* Relational data

Examples:

* PostgreSQL
* MySQL

### NoSQL

Best for:

* Large scale
* Flexible schema
* High throughput

Examples:

* DynamoDB
* Cassandra
* MongoDB

Topics to know:

* Primary Keys
* Secondary Indexes
* Joins
* Denormalization
* Partitioning
* Replication
* Transactions

# 15. Trade-Offs (The Most Important Part)

Every decision has trade-offs.

Common comparisons:

* SQL vs NoSQL
* REST vs gRPC
* Monolith vs Microservices
* Sync vs Async
* Cache vs Database
* Push vs Polling
* WebSocket vs HTTP
* Availability vs Consistency
* Vertical vs Horizontal Scaling

Interviewers care more about *why* than *what*.

# 19. The Interview Framework

Use this checklist in every interview:

Clarify requirements

Estimate scale

Define APIs

Design database

Draw high-level architecture

Explain request flow

Discuss bottlenecks

Add caching

Add message queues

Scale horizontally

Handle failures

Secure the system

Add monitoring

Explain trade-offs

---

# 20. Golden Rules

Ask questions before designing.

Think out loud.

Explain every decision.

Discuss alternatives.

Identify bottlenecks early.

Consider failure scenarios.

Don't over-engineer.

Always explain trade-offs.

There is no perfect design—only designs that fit the requirements.

---

## Final Advice

A great system design interview isn't about building the perfect architecture. It's about demonstrating structured thinking, understanding trade-offs, and communicating clearly.

Remember this simple flow:

**Requirements → Scale → Architecture → APIs → Database → Bottlenecks → Scaling → Reliability → Security → Monitoring → Trade-offs**

Master this framework, and you'll be prepared for the vast majority of senior software engineering system design interviews.
